# 关键技术实现说明文档 (Implementation Details)

## 1. 姿态解算与卡尔曼滤波 (Kalman Filter)
### 1.1 技术难点
*   **加速度计问题**: 动态运动时受重力以外的加速度干扰，数据噪声大，短时不可靠。
*   **陀螺仪问题**: 存在零点漂移，积分运算会导致误差随时间累积，长时不可靠。

### 1.2 解决方案
在 `Middlewares/Algorithm/kalman.c` 中实现了二阶卡尔曼滤波器，融合两者数据：
1.  **预测 (Predict)**: 利用陀螺仪的角速度积分，预测下一时刻的角度。
2.  **更新 (Update)**: 利用加速度计计算出的重力角度作为观测值，修正预测值。

### 1.3 代码实现
```c
// Task_Sensor.c 中的调用
// Delta_t: 采样周期 (0.005s)
// Q_angle: 过程噪声 (0.001)
// R_measure: 测量噪声 (0.03)
Roll = Kalman_GetAngle(&K_Roll, roll_a, gx / 16.4f, Delta_t);
```
此算法成功解决了单纯互补滤波的滞后问题，实现了快速且稳定的姿态跟踪。

## 2. 数据中心 (DataCenter) 与线程安全
### 2.1 设计模式
采用**发布-订阅**模式的简化版。`DataCenter` 作为系统唯一的数据仓库。

### 2.2 线程安全挑战
`Task_Sensor` (写者) 和 `Task_Display` (读者) 并发运行。若不加锁，可能出现"读到一半"的数据（例如：时间从 12:59:59 跳变到 13:00:00 瞬间，可能读出 12:00:00）。

### 2.3 实现机制
使用 FreeRTOS 的 **Mutex (互斥量)**：
```c
void DataCenter_SetSensorData(float roll, ...) {
    if(xSemaphoreTake(SensorMutex, portMAX_DELAY) == pdTRUE) {
        SensorData.Roll = roll;
        // ... 赋值操作
        xSemaphoreGive(SensorMutex);
    }
}
```
这确保了数据的**原子性访问**。

## 3. UI 框架与平滑动画
### 3.1 页面管理
使用**状态机**管理页面 (`UI_Manager.c`)。每个页面封装为三个标准接口：
*   `xxx_Init()`: 初始化数据。
*   `xxx_Draw()`: 绘制逻辑。
*   `xxx_HandleKey()`: 按键回调。

### 3.2 菜单滑动算法
为了实现丝滑的菜单切换，`Page_Menu.c` 引入了**插值动画**：
*   定义 `pre_selection` (当前视觉位置) 和 `target_selection` (目标逻辑位置)。
*   在每次 `Draw` 循环中，计算 `x_pre` (当前 X 坐标) 与目标坐标的差值。
*   每次步进移动 `4px`，直到重合。
*   这使得菜单图标在切换时产生物理惯性般的滑动效果，而非生硬跳变。

## 4. 抬手亮屏算法
### 4.1 逻辑流程
1.  **监测**: `Task_Sensor` 以 200Hz 频率监测 Roll 和 Pitch。
2.  **判定**: 设定"观察姿态"阈值区间（如 Roll: -30°~30°, Pitch: 20°~80°）。
3.  **触发**: 检测到姿态从"非观察区"进入"观察区"的**上升沿**信号。
4.  **执行**: 置位 `IsScreenOn` 标志，重置息屏倒计时。

## 5. 面试问答锦囊 (Q&A Tips)

**Q1: 为什么选择 FreeRTOS 而不是裸机开发？**
> **A**: 项目涉及 UI 刷新、传感器高频采样和复杂算法。裸机的大循环结构难以平衡高实时性（传感器）和高耗时（UI 刷新）任务。FreeRTOS 的**抢占式调度**允许高优先级的 UI 任务打断计算任务，保证了界面的流畅性，同时利用信号量解决了并发冲突。

**Q2: 卡尔曼滤波参数是如何调整的？**
> **A**: 主要调整过程噪声协方差 `Q_angle` 和测量噪声协方差 `R_measure`。增大 `Q_angle` 会让系统更信任陀螺仪（响应快但有漂移风险），增大 `R_measure` 会更信任加速度计（稳定但有滞后）。我在实测中通过串口打印波形，最终找到了兼顾响应速度和稳定性的参数组合。

**Q3: 项目中最大的难点是什么？**
> **A**: 是多任务环境下的**数据一致性**问题。初期偶发显示数据乱码，后来通过引入 `DataCenter` 模块并配合 FreeRTOS 的互斥量机制，规范了所有全局数据的读写权限，彻底解决了竞态条件引发的 Bug。

**Q4: 现在的架构有什么好处？**
> **A**: 现在的架构实现了软硬解耦。如果未来我要把 MCU 换成 ESP32，或者把 MPU6050 换成 MPU9250，我只需要重写 Drivers 层，上层的 App 和 Middlewares 代码几乎不需要改动。本项目已验证了这一点——从 STM32F103 移植到 STM32F407，App 层和 Middlewares 层代码基本未变。

**Q5: F103 到 F407 移植遇到了哪些问题？**
> **A**: 主要是硬件抽象层的 API 差异：GPIO 配置方式不同（单结构体 vs 分离结构体）、RTC 从计数器型变为日历型、SPI 复用引脚需要额外配置、EXTI 中断线配置方式不同等。但由于采用了分层架构，这些变更都被限制在 BSP 驱动层，上层代码几乎不受影响。

## 6. OTA 双分区固件更新系统

### 6.1 系统架构

OTA 系统由三个硬件模块协同工作：
- **W25Q128** (SPI Flash)：固件暂存区，最大 384KB
- **ESP8266** (WiFi)：通过 HTTP GET 从 PC 服务器下载固件
- **AT24C02** (EEPROM)：存储 BootFlags 结构体（启动标志）

### 6.2 BootFlags 数据结构

存储在 AT24C02 的地址 0x00，共 32 字节：
```c
typedef struct {
    uint32_t magic;             // 0x4F544131 ("OTA1")
    uint8_t  active_partition;  // 0=A, 1=B
    uint8_t  ota_pending;       // 1=有待处理的OTA
    uint8_t  ota_target;        // 目标分区 (0=A, 1=B)
    uint8_t  reserved;
    uint32_t fw_length;         // 固件字节数
    uint32_t fw_crc32;          // 固件CRC32校验值
    uint8_t  fw_version[16];    // 固件版本字符串
} BootFlags_t;
```

### 6.3 ESP8266 HTTP 下载流程

1. `ESP8266_Init()` — 初始化 USART3 (115200)，关闭回显
2. `ESP8266_ConnectWiFi()` — `AT+CWJAP` 连接路由器
3. `ESP8266_ConnectTCP()` — `AT+CIPSTART` 建立 TCP 连接
4. `ESP8266_SendData()` — `AT+CIPSEND` + 发送 HTTP GET 请求
5. `ESP8266_ReceiveHTTP()` — 解析 `+IPD,<len>:<data>` 协议，跳过 HTTP header，逐字节写入 W25Q128

### 6.4 固件校验机制

Bootloader 使用 STM32 硬件 CRC 单元计算 CRC32：
```c
RCC_AHB1PeriphClockCmd(RCC_AHB1Periph_CRC, ENABLE);
CRC_ResetDR();
// 逐 256 字节从 W25Q128 读取，按 32 位字送入 CRC 单元
uint32_t crc = CRC_GetCRC();
```

App 端在设置 BootFlags 前使用相同算法计算 CRC32，Bootloader 在烧写前再次计算并比对，确保固件完整性。

### 6.5 固件头验证

STM32F407 的内部 Flash 起始地址为 0x08000000，固件第一个 32 位字是初始栈指针（SP），必须指向 SRAM 范围（0x20000000-0x2001FFFF）：
```c
if (((*(__IO uint32_t *)app_addr) & 0x2FF00000) == 0x20000000) {
    // 有效固件，可以跳转
}
```

### 6.6 Bootloader 跳转机制

```c
static void JumpToApp(uint32_t app_addr)
{
    __disable_irq();
    SysTick->CTRL = 0;              // 停止 SysTick
    __set_MSP(*(__IO uint32_t *)app_addr);  // 设置 MSP
    SCB->VTOR = app_addr;           // 重定向中断向量表
    uint32_t jump_addr = *(__IO uint32_t *)(app_addr + 4);
    __enable_irq();
    ((void(*)(void))jump_addr)();   // 跳转到 App Reset_Handler
}
```

**Q6: OTA 双分区更新有什么优势？**
> **A**: 传统单分区 OTA 如果在写入过程中断电，固件会损坏导致"变砖"。双分区方案中，新固件先下载到外部 Flash，再完整拷贝到非活动分区。整个过程中活动分区的固件始终完好，即使 OTA 失败，重启后仍可运行旧固件。此外，支持版本回退——如果新固件有问题，下次 OTA 可以写回原分区。

**Q7: 为什么用外部 W25Q128 暂存固件，而不是直接写入内部 Flash？**
> **A**: 两个原因：1) 内部 Flash 擦除时不能读取同一 Bank，如果擦除当前运行的分区会导致 HardFault；2) W25Q128 有 16MB 空间，擦写寿命 10 万次，远超内部 Flash 的 1 万次，作为暂存区更安全。
