# STM32+FreeRTOS 智能手表项目 - 技术难点与架构演进文档

本文档记录了本项目在开发过程中遇到的关键技术挑战、架构设计决策以及具体的解决方案。

## 1. 项目概述
本项目基于 STM32F407VET6 微控制器和 FreeRTOS 实时操作系统，构建了一个模块化的智能手表系统。项目集成了 OLED 显示、MPU6050 姿态解算、RTC 时钟以及多个交互式应用（恐龙游戏、秒表等）。

项目最初在 STM32F103C8T6 上开发，后迁移至 STM32F407VET6，获得了更充裕的资源（192KB SRAM、168MHz 主频）和更强的外设（硬件FPU、更多定时器和通信接口）。

## 2. 遇到的棘手问题与技术攻克

### 2.1 UI 系统的上帝对象重构 (Refactoring the "God Object")

**问题描述：**
在项目初期，所有的 UI 绘制逻辑、按键处理逻辑以及应用状态管理都集中在 menu.c 一个文件中。随着功能增加（加入游戏、设置、MPU6050 显示），该文件变得极其臃肿，函数之间耦合度极高。修改一个简单的菜单项可能会意外破坏时钟显示的逻辑，代码维护成本极高。

**技术攻克：Page Controller（页面控制器）模式**
为了解决此问题，我们实施了 UI 架构的模块化重构：
1.  **抽象接口**：定义了统一的页面行为接口。每个页面（Page）必须实现 Draw()（绘制）、HandleKey()（按键处理）和 Reset()（状态重置）函数。
2.  **路由分发**：创建 UI_Manager 模块作为路由器。它维护当前的 CurrentPage 状态，并根据状态将系统事件（刷新、按键）分发给对应的页面控制器。
3.  **物理拆分**：将代码物理拆分为 Page_Clock.c（时钟）、Page_Menu.c（主菜单）、Page_Apps.c（具体应用）。
    *   *效果*：实现了高内聚低耦合，新增一个页面只需增加一个文件并在管理器注册即可，不再影响现有代码。

### 2.2 游戏逻辑与硬件驱动的解耦 (Decoupling Game Logic)

**问题描述：**
恐龙游戏模块 (dino.c) 最初是为了快速实现而编写的，它在游戏的主循环中直接包含了 Key.h 并调用 Key_GetNum() 来检测跳跃动作。
这种设计导致了两个严重问题：
1.  **破坏了事件驱动架构**：游戏绕过了主程序的事件分发机制，直接抢占硬件资源。
2.  **不可移植性**：游戏逻辑与特定的按键驱动绑定，无法轻易移植到其他平台，也无法通过其他方式（如重力感应）控制跳跃。

**技术攻克：依赖倒置与事件注入**
我们对游戏模块进行了手术：
1.  **移除依赖**：从 dino.c 中删除了所有对硬件层 (Key.h) 的引用。
2.  **接口暴露**：游戏模块仅暴露一个控制接口 void Dino_Jump(void)。
3.  **控制反转**：在 Page_Apps.c 的 Page_Game_HandleKey 函数中监听按键事件。当检测到确认键时，由 UI 层调用 Dino_Jump()。
    *   *效果*：游戏引擎变成了纯粹的逻辑计算，不再关心输入是来自按键、触摸屏还是 AI 算法。

### 2.3 FreeRTOS 环境下的隐式声明陷阱 (Implicit Declaration Hell)

**问题描述：**
在重构过程中，我们移除了旧的 menu.h。由于原先 menu.h 为了方便，在内部包含了 LED.h, MyRTC.h, MPU6050.h 等所有外设头文件，main.c 一直依赖这种传递性包含。移除 menu.h 后，编译器在编译 main.c 时报出了大量 Implicit declaration of function 警告，导致链接阶段可能出错或运行时崩溃。

**技术攻克：显式依赖管理**
我们采取了更严格的头文件管理策略：
1.  **显式引入**：在 main.c 中逐一添加了缺失的头文件 (#include "LED.h", #include "MyRTC.h" 等)。
2.  **原则确立**：确立了"谁使用，谁包含"的原则，禁止依赖头文件的嵌套包含来获取函数声明。这极大地提高了代码的可读性和编译的稳定性。

### 2.4 Keil 工程与文件系统的同步 (IDE & Filesystem Sync)

**问题描述：**
我们使用 VS Code 进行代码编辑和重构（移动文件、创建文件夹），但 Keil MDK 的工程文件 (.uvprojx) 不会自动感知文件系统的变化。这导致在 VS Code 中写好的代码，在 Keil 中编译时提示 File not found 或链接错误。

**技术攻克：规范化工程更新流程**
建立了一套标准的操作流程来确保环境同步：
1.  **文件系统重构**：在 VS Code 中建立清晰的目录结构 (App/UI/Pages/)。
2.  **IDE 映射**：在 Keil 中手动创建对应的 Group，并添加新生成的 .c 文件。
3.  **路径配置**：在 Keil 的 C/C++ 选项卡中，将新目录 (.\App\UI\Pages) 添加到 Include Paths 中。

### 2.5 FreeRTOS 调度器启动卡死 (Scheduler Freeze on Startup)

**问题描述：**
在 F103 平台上，系统资源极其紧张（SRAM 仅 20KB），即使将堆栈大小优化到极限，系统在调用 `vTaskStartScheduler()` 后仍然卡死，无法进入第一个任务。调试发现问题出在调度器启动时，无法为 **Idle Task**（空闲任务）和 **Timer Task**（定时器任务）成功分配动态内存。

**技术攻克：静态内存分配 (Static Memory Allocation)**
为了彻底解决启动时的内存分配风险，我们采取了静态分配策略：
1.  **配置变更**：在 `FreeRTOSConfig.h` 中启用 `configSUPPORT_STATIC_ALLOCATION = 1`。
2.  **回调实现**：实现了 `vApplicationGetIdleTaskMemory()` 和 `vApplicationGetTimerTaskMemory()` 接口。
3.  **机制原理**：手动定义全局静态变量作为任务控制块 (TCB) 和栈空间，不再依赖堆 (Heap) 分配器。
    *   *效果*：确保了无论堆内存多么碎片化或紧缺，系统核心任务（Idle/Timer）所需的内存始终是预留且可用的，成功解决了启动卡死问题。

### 2.6 F103 到 F407 的平台移植 (Platform Migration)

**问题描述：**
由于 F103 资源有限（20KB SRAM），系统运行时堆栈空间紧张，限制了功能扩展。决定将项目从 STM32F103C8T6 迁移到 STM32F407VET6，但两者在外设 API、时钟树、GPIO 配置等方面存在显著差异。

**技术攻克：BSP 层逐驱动适配**

#### (1) GPIO 配置差异
F103 使用单结构体配置 GPIO，F407 需要分别配置模式、输出类型和上下拉：
```c
// F103
GPIO_InitStructure.GPIO_Mode = GPIO_Mode_Out_PP;
// F407
GPIO_InitStructure.GPIO_Mode = GPIO_Mode_OUT;
GPIO_InitStructure.GPIO_OType = GPIO_OType_PP;
GPIO_InitStructure.GPIO_PuPd = GPIO_PuPd_UP;
```
GPIO 时钟也从 APB2 改为 AHB1：`RCC_APB2Periph_GPIOx` → `RCC_AHB1Periph_GPIOx`。

#### (2) RTC 从计数器型到日历型
F103 的 RTC 是简单的秒计数器（`RTC_SetCounter`/`RTC_GetCounter`），F407 的 RTC 是完整的日历型，需要使用 `RTC_SetTime()`/`RTC_GetTime()` 操作日历寄存器。时间戳基准值存储在备份寄存器 `RTC_BKP_DR2` 中。

#### (3) SPI 复用引脚
F407 的 SPI 引脚需要额外调用 `GPIO_PinAFConfig()` 配置复用功能，F103 不需要。

#### (4) EXTI 中断配置
F103 使用 `AFIO_EXTILineConfig()`，F407 使用 `SYSCFG_EXTILineConfig()` 并需要使能 SYSCFG 时钟。

#### (5) FreeRTOS 堆大小调整
F407 有 192KB SRAM，将 `configTOTAL_HEAP_SIZE` 从 6KB 增加到 32KB，为任务提供更充裕的内存空间。

#### (6) 系统时钟适配
F407 使用 25MHz HSE 晶振（F103 为 8MHz），PLL 配置为 (25/25)×336/2 = 168MHz。`bsp_delay.c` 的分频系数从 `72/8=9` 改为 `168/8=21`。

*   *效果*：由于采用分层架构，所有变更都限制在 BSP 驱动层，App 层和 Middlewares 层代码基本未改动，仅需将 `#include "stm32f10x.h"` 替换为 `#include "stm32f4xx.h"`。

### 2.7 FreeRTOS Cortex-M4F 移植 — HardFault 崩溃 (FPU Context Corruption)

**问题描述：**
从 F103 迁移到 F407 后，串口显示所有外设初始化成功，但 `vTaskStartScheduler()` 启动后系统崩溃（HardFault）。CFSR 寄存器显示 LSPERR（Lazy State Preservation Error），`pxDelayedTaskList` 指针被破坏为非法地址 `0x10F0F64C`。

**根因分析：**
F103 使用 Cortex-M3（无 FPU），原项目的 FreeRTOS port.c 是 Cortex-M3 版本。F407 使用 Cortex-M4F（有单精度 FPU），编译器默认使用 FPU 指令。Cortex-M3 版 port.c 的 PendSV handler 不保存/恢复 FPU 寄存器（S16-S31），导致：
1. 任务切换时 FPU 寄存器被破坏
2. FPU lazy stacking 触发 LSPERR 总线错误
3. FreeRTOS 内部数据结构（如 pxDelayedTaskList）被 FPU 栈溢出覆盖

**技术攻克：使用官方 RVDS/ARM_CM4F 端口**
从 GitHub FreeRTOS-Kernel 仓库下载官方 RVDS/ARM_CM4F 端口，关键差异：

1. **PendSV handler 保存 FPU 上下文**：
```asm
tst r14, #0x10          /* 检查 EXC_RETURN bit 4 */
it eq
vstmdbeq r0!, {s16-s31} /* 如果 FPU 上下文活跃，保存 S16-S31 */
stmdb r0!, {r4-r11, r14} /* 保存核心寄存器 + EXC_RETURN */
```

2. **SVC handler 从栈恢复 EXC_RETURN**：
```asm
ldmia r0!, {r4-r11, r14} /* 恢复核心寄存器 + EXC_RETURN */
bx r14                    /* 用恢复的 EXC_RETURN 返回 */
```

3. **初始化栈帧存储 EXC_RETURN**：
```c
*pxTopOfStack = portINITIAL_EXC_RETURN; /* 0xFFFFFFFD */
```

4. **FPU 使能**：
```asm
/* prvEnableVFP: 使能 CP10/CP11 协处理器 */
orr r1, r1, #(0xf << 20)
str r1, [r0]
/* prvStartFirstTask: 清除 CONTROL.FPCA 位 */
mov r0, #0
msr control, r0
```

5. **FPCCR 配置**：
```c
*(portFPCCR) |= portASPEN_AND_LSPEN_BITS; /* 使能 FPU lazy stacking */
```

*   *效果*：使用官方端口后，FreeRTOS 调度器正常启动，任务切换稳定，FPU 上下文正确保存/恢复。

## 3. 架构现状
经过上述改进和平台迁移，目前的系统架构如下：

*   **硬件抽象层 (HAL/Drivers)**: bsp_key, bsp_oled, bsp_mpu6050, bsp_led, bsp_usart, bsp_delay, bsp_timer, bsp_rtc, bsp_adc (负责底层寄存器操作)
*   **操作系统层 (OS)**: FreeRTOS ARM_CM4F (负责任务调度)
*   **应用框架层 (Framework)**: UI_Manager (负责页面路由), App_Tasks (负责业务流程)
*   **业务逻辑层 (Business Logic)**: Page_Clock, Page_Menu, Page_Apps, dino (纯业务逻辑)

## 4. OTA 双分区固件更新系统

### 4.1 系统概述

为 SmartWatch 增加 OTA（Over-The-Air）无线固件更新功能。硬件基础：W25Q128（SPI Flash，固件暂存）、ESP8266（WiFi，HTTP下载）、AT24C02（EEPROM，Boot标志存储）。实现真双 A/B 分区更新，断电安全，可回退。

### 4.2 分区设计

```
STM32F407VE 内部 Flash (512KB):
┌──────────────────────────────────────────────────┐
│ Sector 0-3  │ 0x08000000-0x0800FFFF │  64KB     │ Bootloader
├──────────────────────────────────────────────────┤
│ Sector 4    │ 0x08010000-0x0801FFFF │  64KB     │ App_A
│ Sector 5    │ 0x08020000-0x0803FFFF │ 128KB     │ App_A (共192KB)
├──────────────────────────────────────────────────┤
│ Sector 6    │ 0x08040000-0x0805FFFF │ 128KB     │ App_B
│ Sector 7    │ 0x08060000-0x0807FFFF │ 128KB     │ App_B (共256KB)
└──────────────────────────────────────────────────┘
```

OTA 流程：当前运行 App_A → ESP8266 下载新固件到 W25Q128 → 设置 ota_pending 标志 → 重启 → Bootloader 从 W25Q128 拷贝固件到分区 B → 跳转执行。

### 4.3 Bootloader 中 printf 完全失效

**问题描述：**
Bootloader 工程使用 `printf` 输出调试信息，但串口只输出乱码或完全无输出。即使包含 `<stdio.h>` 并重定向 `fputc`，`printf` 仍然不工作。

**根因分析：**
Bootloader 是裸机工程（无 FreeRTOS），C 库的 `printf` 依赖 semihosting 或 syscall 机制。在 Keil 裸机工程中，`printf` 需要：
1. 正确的 heap 分配（`__microlib` 或手动实现 `malloc`）
2. 半主机模式（semihosting）需要调试器连接
3. `fputc` 重定向只在使用 MicroLIB 时生效

这些条件在 Bootloader 跳转场景下不满足，导致 `printf` 调用卡死或产生 HardFault。

**技术攻克：完全绕过 C 库，直接操作 USART 寄存器**

删除所有 `printf` 调用，用底层 USART 寄存器操作替代：
```c
static void uart_putc(char c)
{
    while ((USART1->SR & 0x40) == 0);  // 等待发送完成
    USART1->DR = c;
}

static void uart_puts(const char *s)
{
    while (*s) uart_putc(*s++);
}

static void uart_hex32(uint32_t v)
{
    const char h[] = "0123456789ABCDEF";
    uart_puts("0x");
    for (int i = 28; i >= 0; i -= 4)
        uart_putc(h[(v >> i) & 0x0F]);
}

static void uart_uint(uint32_t v)
{
    char buf[11];
    int i = 0;
    if (v == 0) { uart_putc('0'); return; }
    while (v > 0) { buf[i++] = '0' + (v % 10); v /= 10; }
    while (i > 0) uart_putc(buf[--i]);
}
```

*效果*：完全消除对 C 库的依赖，输出可靠且零开销。

### 4.4 SysTick 中断干扰 USART1 发送

**问题描述：**
Bootloader 串口输出在 `SysTick_Init()` 之后截断。例如：
```
[BOOT] Bootloader v1.0
[BOOT] STM32F407VE OTA
========          ← 此处截断，后续全部丢失
```
只输出前两行就卡死，或者输出乱码。

**根因分析：**
`bsp_delay.c` 中 `SysTick_Init()` 使能了 SysTick 中断（`TICKINT=1`）：
```c
SysTick->CTRL |= (1 << 1);  // TICKINT=1，使能中断
SysTick->CTRL |= (1 << 0);  // ENABLE=1
```
Bootloader 没有实现 `SysTick_Handler`（或使用空实现），但 SysTick 中断触发时会：
1. 打断 `uart_putc` 的 `while ((USART1->SR & 0x40) == 0)` 等待循环
2. 中断返回后 USART 状态可能已变化
3. 累积效应导致后续所有输出失败

**技术攻克：禁用 SysTick 中断，使用纯轮询模式**

修改 `bsp_delay.c`，去掉 `TICKINT` 位：
```c
void SysTick_Init(uint16_t sysclk)
{
    SysTick->CTRL = 0;
    SysTick->CTRL |= (1 << 2);  // CLKSOURCE=1，处理器时钟
    g_fac_us = sysclk;
    SysTick->LOAD = 0xFFFFFF;   // 最大重载值
    SysTick->VAL  = 0;
    SysTick->CTRL |= (1 << 0);  // ENABLE=1，不使能中断 (无 TICKINT)
}
```

*效果*：`Delay_us()` 和 `Delay_ms()` 通过轮询 SysTick 计数器工作，不依赖中断，USART1 输出稳定可靠。

### 4.5 Keil Flash 下载地址错误导致 Bootloader 被覆盖

**问题描述：**
烧录 App 后 Bootloader 消失，串口只输出 `========`（Bootloader 的 uart_puts 输出不完整或被破坏）。重新烧录 Bootloader 后又正常，但再次烧录 App 又消失。

**根因分析：**
Keil 的 Flash Download 配置默认从 `0x08000000` 开始烧录整个芯片。App 工程的 `.sct` 散列文件已改为 `0x08010000`，但 Keil 的 **Flash Download** 设置（Options → Utilities → Settings）未同步修改：
- `Start`: 仍为 `0x08000000`
- `Size`: 仍为 `0x00080000`（512KB 全片）

这导致烧录 App 时，Keil 从 `0x08000000` 开始擦写，覆盖了 Bootloader 所在的 Sector 0-3。

**技术攻克：修改 Flash Download 配置**

在 Keil 中修改两个工程的 Flash Download 设置：

**Bootloader 工程：**
- Start: `0x08000000`
- Size: `0x10000`（64KB，Sector 0-3）

**App 工程：**
- Start: `0x08010000`
- Size: `0x30000`（192KB，Sector 4-5）

*效果*：两个工程独立烧录，互不覆盖。

### 4.6 ESP8266 AT 指令响应中 "OK" 被拆分

**问题描述：**
发送 `AT\r\n` 后 ESP8266 有响应（收到多个字节），但代码检测不到 "OK"。串口打印显示收到的数据为：
```
Hex: 41 54 0D 0A 0D 0A 4F 4B 0D 0A
Chr: [AT<0D><0A><0D><0A>OK<0D><0A>]
```

**根因分析：**
ESP8266 回显 `AT` 后返回 `\r\nOK\r\n`。早期检测代码使用连续字符匹配：
```c
if (resp[i] == 'O' && resp[i+1] == 'K') { found_ok = 1; }
```
但某些固件版本会在响应中插入额外的 `\r\n`，导致 `O` 和 `K` 不连续。

**技术攻克：改为子串搜索**

在 `esp8266.c` 中使用 `strstr()` 进行鲁棒的子串匹配：
```c
if (strstr(esp_buf, target) != NULL) {
    return 0;  // 找到目标字符串
}
```

*效果*：无论 ESP8266 响应格式如何变化（有无回显、有无额外换行），都能正确识别。

### 4.7 HTTP 服务器端口冲突导致 OTA 下载失败

**问题描述：**
OTA 下载固件时，ESP8266 连接服务器成功，但收到的是 HTML 错误页面（404）而非固件数据。浏览器直接访问 `http://192.168.1.3:8080/SmartWatch_F407.bin` 也返回 404。

**根因分析：**
PC 上端口 8080 已被其他程序占用（`Embedthis-http`，进程 PID 11952）。`python -m http.server 8080` 虽然启动成功，但实际监听失败或被抢占，请求被路由到了错误的服务。

**技术攻克：更换端口**

通过 `netstat -ano | findstr :8080` 发现端口冲突后：
1. 将 OTA 服务器端口从 8080 改为 9090：`ota_manager.h` 中 `#define OTA_SERVER_PORT 9090`
2. Python HTTP 服务器改用 9090 端口：`python -m http.server 9090`
3. 浏览器验证 `http://192.168.1.3:9090/SmartWatch_F407.bin` 可正常下载

*效果*：OTA 固件下载正常完成。

### 4.8 Keil 工程 Flash 驱动不匹配

**问题描述：**
Bootloader 烧录后串口无任何输出，芯片似乎完全不运行。

**根因分析：**
Keil 的 Flash 编程算法选择错误。Bootloader 工程使用了 `STM32F4xx_1024`（1MB Flash 驱动），但 STM32F407VE 只有 512KB Flash，应使用 `STM32F4xx_512` 驱动。错误的驱动导致 Flash 编程时序不正确，写入的数据损坏。

**技术攻克：修正 Flash 编程算法**

在 Keil → Options → Utilities → Settings → Flash Download 中：
- 将 Programming Algorithm 从 `STM32F4xx 1024KB` 改为 `STM32F4xx 512KB`

*效果*：Flash 烧录正确，Bootloader 正常运行。

### 4.9 StartTask 堆栈溢出

**问题描述：**
App 工程启动后 HardFault 崩溃，或 FreeRTOS 报 stack overflow。

**根因分析：**
`task_manager.c` 中的 `StartTask` 堆栈大小设置过小（80 words = 320 字节）。StartTask 调用了多个测试函数（`Test_W25Q128`、`Test_AT24C02`、`Test_ESP8266_AT`、`Test_OTA_Full`），每个函数内部都有局部缓冲区（如 `uint8_t r_buf[16]`、`char cmd[128]`），总栈消耗超过 320 字节。

**技术攻克：增大堆栈**
```c
// app_config.h
#define TASK_STACK_SIZE_START  512  // 从 80 → 256 → 512 words (2048 字节)
```

*效果*：所有测试函数正常运行，无溢出。
