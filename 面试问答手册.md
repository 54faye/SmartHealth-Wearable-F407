### Q4: 异步SD NAND日志模块是怎么设计的？为什么要做异步？

**痛点：** SD卡写入是耗时操作（涉及SPI通信 + 文件系统操作 + Flash编程），一次写入可能耗时几毫秒到几十毫秒。如果在业务线程（如动画渲染、音频处理、MQTT通信）中同步写日志，会直接阻塞高优先级任务，导致动画卡顿、音频断续。这是我在实际测试中遇到的问题——加了同步日志后动画帧率从30fps掉到十几fps。

**解决方案：** 设计了异步非阻塞的日志架构：

1. **FreeRTOS队列接收：** 业务线程调用`sd_log_error(tag, code, msg)`，日志条目入队（`xQueueSend`超时为0，不阻塞），立即返回。队列深度80条，队列满时丢弃并计数（`std::atomic<uint32_t>`丢弃计数器）。
2. **PSRAM缓冲零占用SRAM：** 队列存储区和写入缓冲区都用`EXT_RAM_BSS_ATTR`宏声明，编译器将这些大数组（80条×104字节队列 + 8KB写缓冲）放到PSRAM的BSS段，不占用宝贵的片内SRAM（ESP32-S3内部SRAM只有512KB，DMA和任务栈都需要用）。队列控制块`StaticQueue_t`（48字节）留在内部RAM，因为FreeRTOS内核频繁访问它。
3. **BackgroundTask调度刷盘：** 不用专门的写日志任务，而是复用Application的BackgroundTask（一个通用的后台任务调度器，内部用`std::condition_variable`唤醒）。定时器每500ms触发检查，或缓冲区80%满时立即调度。
4. **双阈值刷盘设计：**
   - **低阈值（时间触发）：** 距上次刷盘≥500ms且有数据就刷——保证日志不会在缓冲区停留太久，即使日志量小也能及时落盘。
   - **高阈值（容量触发）：** 缓冲区使用率≥80%立即刷——日志突发时快速清空缓冲区，防止队列积压导致丢弃。
   - 两个阈值是"或"关系，满足任一即触发。500ms是经验值，平衡了写入频率和数据安全性；80%留了20%余量，防止刷盘过程中新日志把缓冲区写满。

**文件轮转逻辑：**
- 单文件最大512KB（`SD_LOG_FILE_MAX_SIZE`），写满后创建新文件（不重命名旧文件，减少文件系统操作）。
- 最多保留10个文件（`SD_LOG_MAX_FILES`），超出时删除最旧的。清理操作分段持锁：先持锁收集文件列表（~10ms），锁外排序（不阻塞动画），再每个文件单独持锁删除（中间可插队，减少对动画的阻塞）。
- 文件名格式：时间已同步时`error_YYYYMMDD_HHMMSS_XXX.log`（序号同秒递增，跨秒重置），时间未同步时`error_nosync_XXX.log`（全局序号递增）。重启时扫描现有文件，复用最新且未写满的文件。

**下载保持模式：** 动画资源从MQTT批量下载时，SD卡被下载任务占用，此时`sd_log_enter_download_mode()`禁止日志刷盘，缓冲区满就丢弃；下载结束后`sd_log_exit_download_mode()`立即触发补刷盘。

### Q5: 0xMMEE结构化错误码是怎么设计的？覆盖了哪些模块？

**格式：** `0xMMEE`，高8位是模块编号（MM），低8位是具体错误类型（EE）。用宏`MAKE_LOG_ERR(module, err_id)`生成，如`MAKE_LOG_ERR(0x06, 0x01)` = `0x0601`（IMU模块初始化失败）。

**为什么这么设计：**
1. **可机读：** 运维工具可以按模块编号批量过滤/统计错误，不需要解析自然语言。
2. **可扩展：** 一个模块最多256个错误码，16位空间足够覆盖所有子模块。
3. **与日志格式配合：** 日志行格式为`[E] 2025-01-15 10:30:45 code=0x0601 [imu] IMU 初始化失败`，错误码和人类可读消息并存。

**覆盖的模块（共18个）：**

| 模块编号 | 模块名 | 典型错误码 |

| 0x01 | Animation | 加载失败、QOI解码失败、PSRAM分配失败 |

| 0x03 | Battery | 低电压、充电异常、UART CRC失败 |
| 0x04 | BLE | 连接失败、配对错误、GATT服务错误 |

### Q6: 两阶段崩溃溯源是怎么设计的？为什么分两个阶段？

**核心问题：** 设备在用户手上崩溃了，怎么远程获取崩溃现场？

**设计原则：Panic阶段不写SD卡。** 原因是崩溃时系统状态不确定——SPI总线可能处于中间态、文件系统可能未flush、SD卡可能正在写入。如果在Panic Handler中直接操作SD卡，极大概率触发二次异常（总线超时、死锁），反而丢失原始崩溃信息。

**两阶段设计：**

**第一阶段：崩溃时（Panic Handler）—— 只写Flash**
- ESP-IDF的`esp_core_dump`机制在Panic Handler中自动将CPU寄存器、任务栈、RTOS任务列表等关键信息写入Flash的coredump分区（专用的Data分区）。
- 这个过程是ESP-IDF框架自动完成的，代码量极小，不涉及文件系统和SPI驱动，可靠性极高。
- 禁用`NO_OVERWRITE`标志，每次崩溃都覆盖写入，确保最新的崩溃现场被捕获。

**第二阶段：重启后（系统稳定时）—— Flash搬运到SD卡**
- `sd_crash_log_init()`在启动早期调用，从NVS读取并递增boot_count。
- `sd_crash_log_recover()`在SD卡挂载成功后调用：
  1. 调用`esp_core_dump_image_check()`检查Flash中是否有有效的coredump。
  2. 读取coredump的Flash地址和大小（`esp_core_dump_image_get`）。
  3. 在SD卡创建crash目录（`/sdcard/logs/crash/crash_YYYYMMDD_HHMMSS/`）。
  4. 写入`summary.log`：包含reset reason（PANIC/WDT/BROWNOUT等15种）、固件版本、时间戳、boot_count、coredump大小。
  5. 写入`coredump.bin`：从Flash分区按512字节分块读取（`COREDUMP_READ_BUF_SIZE`），逐块写入SD卡。写入时保留完整的24字节core_dump_header_t，因为`espcoredump.py`工具需要这个头部进行校验和ELF定位。
  6. 清理旧crash目录（最多保留5个）。

**为什么不会在搬运过程中再次崩溃？**
- 重启后系统是全新状态，SPI总线空闲，文件系统已正确挂载，没有并发访问。
- NVS记录了boot_count，如果连续崩溃（boot_count快速增长），可以通过分析crash目录的时间戳和reset reason来判断是启动阶段崩溃还是运行时崩溃。

### Q7: A/B分区OTA升级方案是怎么设计的？和你个人项目的OTA有什么区别？

**实习项目（ESP32-S3）的OTA：**

基于ESP-IDF的`esp_ota_ops` API实现的流式OTA：
1. **版本检查：** `CheckVersion()`向OTA服务器POST设备信息（MAC、UUID、序列号），服务器返回JSON包含firmware版本、URL、MQTT配置、WebSocket配置、server_time等。版本比较用`IsNewVersionAvailable()`逐段解析`major.minor.patch`。
2. **流式下载：** `Upgrade()`通过HTTP GET下载固件，边下载边写入OTA分区（`esp_ota_write`），不需要先把整个固件下载到RAM或外部Flash。每秒计算一次下载速度和进度，更新Display状态栏。
3. **镜像校验：** 下载开始时先积累足够的头部数据（`esp_image_header_t` + `esp_image_segment_header_t` + `esp_app_desc_t`），验证新固件版本号与当前不同（防止重复升级）。`esp_ota_end()`会做完整的镜像校验（SHA-256 + 签名）。
4. **启动标记：** `esp_ota_set_boot_partition()`设置下次启动分区。重启后新固件进入`ESP_OTA_IMG_PENDING_VERIFY`状态，运行一段时间后调用`MarkCurrentVersionValid()`确认有效（`esp_ota_mark_app_valid_cancel_rollback()`），否则ESP-IDF的Rollback机制会自动回滚到旧分区。
5. **设备激活：** 对有序列号的设备，OTA时还支持HMAC-SHA256挑战-响应激活机制（`GetActivationPayload()`），使用eFuse中的Key0计算HMAC，防止未授权设备使用。

**和个人项目（STM32F407）OTA的关键区别：**

| 维度 | 实习项目（ESP32） | 个人项目（STM32） |
|------|-----------------|-----------------|
| 下载方式 | 流式写入OTA分区 | 先下载到W25Q128外部Flash |
| 镜像校验 | ESP-IDF内置SHA-256 + 签名 | 手动CRC32（硬件外设） |
| 回滚机制 | ESP-IDF自动Rollback（PENDING_VERIFY） | 手动A/B分区 + BootFlags |
| 安全性 | HMAC设备激活 + 签名校验 | 仅magic number防随机数据 |
| 分区管理 | ESP-IDF partition table自动管理 | 手动定义Flash地址 |
| 复杂度 | 框架级，调用API即可 | 从零实现Bootloader跳转逻辑 |

实习中的OTA是工业级的，依赖ESP-IDF框架的安全机制；个人项目是从零实现的，更深入理解了Bootloader、分区管理、CRC校验的底层原理。

### Q8: 你实习中遇到过哪些Bug？怎么排查的？

**案例一：SD卡写入偶尔失败**

- **现象：** 日志模块上线后，偶尔出现`SD_ERR_WRITE_FAILED`（0x0A02），频率约千分之一。
- **复现：** 高频日志 + 动画资源下载同时进行时更容易复现——两个任务竞争SD卡访问。
- **定位：** 通过逻辑分析仪抓SPI波形，发现写入失败时CS（片选）信号在一次写入过程中被拉高了——原因是sdcard模块的互斥锁被动画下载任务抢占后释放，导致CS信号异常。
- **解决：** 重新设计了SD卡锁的粒度——将"文件操作"级别的锁改为"SPI事务"级别的锁，确保一次完整的SPI写入不会被打断。同时在日志模块中增加了下载保持模式（`sd_log_enter_download_mode`），下载期间禁止日志刷盘。
- **验证：** 修改后连续运行72小时未出现写入失败。

**案例二：OTA升级后设备反复重启**

- **现象：** 某次OTA升级后设备进入重启循环，每次启动几秒后就重启。
- **定位：** 通过串口日志发现reset reason是`ESP_RST_TASK_WDT`（任务看门狗超时），进一步分析coredump发现是MQTT任务在启动阶段卡死——原因是OTA服务器返回的MQTT配置JSON中缺少了必填字段，导致MQTT连接初始化时访问了空指针。
- **解决：** 1) 在MQTT配置解析时增加空指针检查和默认值；2) 利用ESP-IDF的Rollback机制——新固件进入`PENDING_VERIFY`状态后，如果在确认窗口期内没有调用`MarkCurrentVersionValid()`，系统自动回滚到旧固件。
- **验证：** 修复后故意用错误配置测试，设备成功回滚到旧版本。

**案例三：PSRAM分配失败导致动画崩溃**

- **现象：** 长时间运行后动画模块偶尔崩溃，错误码`ANIM_ERR_PSRAM_ALLOC`（0x010B）。
- **定位：** 用`heap_caps_get_free_size(MALLOC_CAP_SPIRAM)`监控PSRAM使用量，发现随着时间推移PSRAM逐渐减少——原因是动画资源下载后缓存在PSRAM中，但某些错误路径下没有正确释放。
- **解决：** 用RAII包装PSRAM分配（`PsramAllocator<T>` + `std::unique_ptr`），确保异常路径也能正确释放。同时增加了PSRAM使用量的运行时监控告警。

### Q9: 你的Debug工作流是什么？

我的标准排查流程：

1. **复现问题：** 先确认能稳定复现，记录复现步骤和环境条件（WiFi信号强度、SD卡类型、电量等）。
2. **添加日志定位模块：** 用`sd_log_error()`记录关键路径的状态，结合0xMMEE错误码快速定位是哪个模块出了问题。
3. **用调试工具分析：**
   - **串口日志（ESP_LOGI/E）：** 第一时间获取调用栈和错误信息。
   - **逻辑分析仪：** 抓SPI/I2C/UART波形，排查通信时序问题。
   - **J-Link + GDB：** 设断点、查看内存、分析coredump。
   - **`idf.py size`：** 分析固件大小，确认是否超出分区限制。
4. **提出解决方案：** 根据根因设计修复方案，考虑边界情况和副作用。
5. **验证并回归测试：** 修复后不仅验证原问题不复现，还要回归相关功能。
6. **提交PR并编写文档：** commit message描述问题现象、根因、修复方案；如果是架构性问题，在飞书文档中记录设计决策。

### Q10: 项目中ESP32-S3的PSRAM是怎么用的？

ESP32-S3外挂了Octal SPI PSRAM（80MHz），项目中的关键用途：
1. **日志队列存储区：** `EXT_RAM_BSS_ATTR static uint8_t s_queue_storage[80 * 104]`——80条日志条目，约8KB，放在PSRAM。
2. **日志写入缓冲区：** `EXT_RAM_BSS_ATTR static char s_write_buffer[8192]`——8KB批量刷盘缓冲区，放在PSRAM。
3. **动画帧缓冲：** 360x360 LCD的LVGL draw buffer分配在PSRAM中。
4. **自定义分配器：** `PsramAllocator<T>`模板类优先从PSRAM分配，失败再回退到内部SRAM。
5. **内存规划：** 内部SRAM（512KB）保留给DMA描述符、FreeRTOS任务栈、中断处理等对延迟敏感的用途；大容量数据（日志、动画、音频缓冲）放PSRAM。

### Q11: OPUS编码器由C到C++的重构封装，具体怎么做的？

OPUS是开源音频编码器，原始API是纯C风格（`opus_encoder_create`、`opus_encode`等）。封装为C++类`OpusEncoderWrapper`（在`components/78__esp-opus-encoder/opus_encoder.cc`中）：

1. **RAII封装：** 构造函数创建编码器实例，析构函数释放，避免资源泄漏。
2. **接口简化：** 将`opus_encode(enc, pcm, frame_size, data, max_data_bytes)`封装为`Encode()`方法，内部处理采样率、通道数、帧长等参数。
3. **与ESP-IDF音频pipeline集成：** 作为`AudioCodec`的子组件，配合I2S驱动实现音频的实时编码。
4. **重采样支持：** `opus_resampler.cc`处理输入采样率与OPUS要求的采样率（如16kHz）之间的转换。

### Q12: 项目中FreeRTOS的任务架构是怎样的？

项目运行在ESP32-S3的双核上，主要任务：
- **Main Task（Core 0）：** Application主循环，管理设备状态机（Idle/Listening/Speaking等17+状态）。
- **Audio Task：** 音频采集 + OPUS编码 + 网络发送，对实时性要求最高。
- **Display Task：** LVGL渲染，50ms刷新周期。
- **Background Task：** 通用后台调度器（`BackgroundTask`类），用`std::condition_variable`唤醒，日志刷盘、资源下载等非实时操作都在这里执行。
- **MQTT Task：** 设备管理通道，处理远程控制命令、OTA、资产同步。

任务间同步用FreeRTOS队列（音频数据）、互斥锁（SD卡、LVGL）、事件组（游戏状态）。

### Q13: 项目的通信协议是怎么设计的？

项目支持两种通信协议：

### Q15: 多级低功耗策略具体是怎么实现的？

**痛点：** 智能手表用300mAh锂电池，如果不做低功耗优化，全速运行只能撑十几个小时。

**四级低功耗策略（从底到顶）：**

**第一级：CPU级 —— WFI指令**
- `vApplicationIdleHook()`中调用`__WFI()`（Wait For Interrupt），当所有任务都在阻塞时，CPU进入Sleep模式，等待中断唤醒。
- WFI只暂停CPU内核，外设（I2C、SPI、UART）和NVIC中断控制器仍正常工作。SysTick中断每1ms唤醒一次CPU，但如果没有任何就绪任务，CPU立即再次进入WFI。
- **收益：** CPU不再空转烧功耗，待机时CPU功耗从~30mA降至~5mA（STM32F407 Sleep模式）。

**第二级：外设级 —— OLED自动熄屏 + 降频刷新**
- `DisplayTask`中检测`LastActionTime`（最后一次按键或运动的时间），3分钟（`OLED_AUTO_OFF_TIMEOUT`）无操作后：
  1. 调用`OLED_Clear()` + `OLED_Update()`清屏，OLED进入休态（SSD1306休眠功耗~10μA）。
  2. DisplayTask不再调用`vTaskDelayUntil`（50ms周期），改为`vTaskDelay(500ms)`轮询——**刷新频率从20fps降到2fps**，减少10倍的I2C通信和CPU唤醒次数。
- **实现关键：** 用`if (!IsScreenOn) { vTaskDelay(500ms); continue; }`跳过正常的刷新流程，同时保持任务存活以便快速恢复。
- **收益：** OLED熄屏后功耗降低约15mA（OLED全屏点亮约15-20mA），加上降频刷新减少CPU唤醒，整体待机电流进一步降低。

**第三级：传感器级 —— 自适应采样率**
- MPU6050Task在每次循环末尾判断当前页面：如果屏幕亮着且当前页面是`PAGE_MPU6050`或`PAGE_GRADIENTER`（需要实时数据），采样周期设为5ms（200Hz）；否则设为100ms（10Hz）。
- **为什么这样设计：** 200Hz采样时MPU6050功耗约3.4mA，10Hz时可以忽略不计。大部分时间用户只是看时钟，不需要高频采样。
- **切换时不会丢数据：** 切换发生在当前循环结束后，卡尔曼滤波器的`dt`参数也同步调整（`Delta_t = 0.005f`或`0.1f`），保证融合算法的正确性。
- **收益：** 传感器功耗从常驻3.4mA降至大部分时间~0mA，仅在查看传感器页面时短暂提升。

**第四级：交互级 —— 抬腕唤醒**
- 不需要按键就能点亮屏幕，降低交互门槛。
- 具体实现在Q29中详细说明。

**总体效果：** 优化前待机电流约20mA（CPU全速 + OLED常亮 + 传感器200Hz），优化后约6mA（CPU WFI + OLED熄屏 + 传感器10Hz），**降低约70%**。300mAh电池续航从~15小时提升到~50小时。

### Q16: 抬腕检测的算法是怎么实现的？怎么避免误触发？

**原理：** 用卡尔曼滤波融合后的Roll/Pitch角度判断手腕姿态。当Roll在±45°、Pitch在-45°~+10°范围内时，认为是"抬手看表"姿势。

**为什么用融合角而不是原始数据：** 原始加速度计噪声大，直接判断会导致频繁误触发。卡尔曼滤波后的角度平滑稳定，判断阈值可以设得更精确。

**边沿触发设计（核心防误触机制）：**
- 用`last_posture_ok`变量记录上一次的姿态状态。
- **只在从"不符合"到"符合"的状态跳变时触发亮屏：** `if (current_posture_ok && !last_posture_ok)`
- 如果用户一直保持抬手姿势，屏幕只亮一次，不会反复触发。
- 放下手后`last_posture_ok`被清零，再次抬起时才能重新触发。

**屏幕亮时的状态维护：** 屏幕亮着时也实时更新`last_posture_ok`，确保用户放下手（状态从1变0）再抬起时能正确检测到上升沿。如果不维护这个状态，放下手后`last_posture_ok`还保持为1，再次抬起时就不会触发。

**运动检测防熄屏：** `task_sensor.c`中还检测Roll/Pitch的变化量（`fabs(Roll - last_Roll) > MOVEMENT_THRESHOLD`），只要手腕在动就刷新`LastActionTime`，防止OLED在运动中自动熄屏。

### Q17: OTA方案的整体架构是什么？和实习中的OTA有什么区别？

**本项目（STM32F407）的OTA——从零实现：**

A/B双分区 + 外部暂存的经典方案：

1. **分区布局：** Bootloader（0x08000000, 64KB）+ App_A（0x08010000, 192KB）+ App_B（0x08040000, 256KB）
2. **下载阶段：** App通过ESP8266 WiFi从HTTP服务器下载固件到W25Q128外部Flash（作为暂存区）
3. **校验阶段：** 用STM32硬件CRC32外设计算固件校验和，与服务器下发的CRC比对
4. **标记阶段：** 校验通过后，将BootFlags（magic=0x4F544131"OTA1"、ota_pending=1、ota_target=非活动分区、fw_crc32、fw_length）写入AT24C02 EEPROM
5. **升级阶段：** 系统复位，Bootloader读取BootFlags，从W25Q128搬运固件到目标分区，搬运前后双重CRC校验
6. **启动阶段：** 校验通过后设置active_partition，跳转App

**为什么用W25Q128暂存而不直接写片内Flash？**
- ESP8266下载速度不确定（WiFi可能断断续续），如果边下载边写片内Flash，中途断开会导致Flash处于不一致状态——新固件写了一半，旧固件可能已被擦除。
- 片内Flash擦除粒度大（最小16KB扇区），下载到一半失败后重试代价大。
- W25Q128是16MB SPI NOR Flash，4KB扇区擦除快，下载和升级解耦——可以先下载完、校验通过，再决定是否升级。

**和实习中ESP32 OTA的关键区别：**

| 维度 | 实习项目（ESP32） | 本项目（STM32） |
|------|-----------------|-----------------|
| 下载方式 | 流式写入OTA分区（`esp_ota_write`） | 先下载到W25Q128外部Flash |
| 镜像校验 | ESP-IDF内置SHA-256 + 签名 | 手动硬件CRC32（`RCC_AHB1Periph_CRC`） |
| 回滚机制 | ESP-IDF自动Rollback（PENDING_VERIFY） | 手动A/B分区 + BootFlags |
| 安全性 | HMAC设备激活 + 签名校验 | magic number防随机数据 + SP指针校验 |
| 复杂度 | 框架级，调用API即可 | 从零实现Bootloader跳转逻辑 |

### Q18: 如果OTA升级过程中突然断电了怎么办？

**不会变砖，这是A/B分区设计的核心目标。** 具体分析每个阶段断电的后果：

| 断电时机 | 系统状态 | 后果 |
|---------|---------|------|
| 下载阶段（正在写W25Q128） | BootFlags未设置，ota_pending=0 | Bootloader直接跳转当前活动分区，正常运行 |
| 校验阶段 | 固件不完整，CRC不匹配 | 不会设置BootFlags，同上 |
| 标记阶段（正在写EEPROM） | BootFlags可能写了一半 | magic字段可能不完整，Bootloader检测到magic≠0x4F544131，跳过升级 |
| Bootloader搬运阶段 | 固件正在写入非活动分区 | Bootloader检测到ota_pending=1但CRC不匹配（搬运不完整），不切换active_partition |
| 搬运完成但未设置active | 新分区已写入，但active还指向旧分区 | Bootloader验证新分区CRC通过后才设置active，如果断电在设置之前，下次启动仍用旧分区 |

**关键设计：Bootloader在任何环节出错都会回退到当前活动分区。** BootFlags的magic字段（0x4F544131 = ASCII "OTA1"）是第一道防线——AT24C02上电后内容是随机的，如果magic不对就直接跳过。

### Q19: Bootloader跳转App前做了哪些安全检查？

Bootloader是系统可靠性的最后一道防线，跳转前做6项检查：

1. **BootFlags有效性：** 读取AT24C02中的BootFlags，检查magic是否为0x4F544131。不是则直接跳转默认分区。
2. **ota_pending标志：** 如果ota_pending=1，需要先完成固件搬运和CRC校验。
3. **搬运前后双重CRC：** 从W25Q128按256字节分块读取固件，送入STM32硬件CRC单元（`RCC_AHB1Periph_CRC`，168MHz下每个时钟周期处理一个32-bit字，192KB只需~0.3ms），与BootFlags中的fw_crc32比对。搬运后再对片内Flash做一次CRC校验，确保搬运过程无损。
4. **栈指针校验：** 读取App起始地址的第一个字（初始SP），检查是否指向SRAM区域（`(SP & 0x2FF00000) == 0x20000000`）。如果App未烧写或损坏，这个值会是0xFFFFFFFF（全F是Flash擦除后的默认值），校验失败不会跳转。
5. **中断/SysTick关闭：** `__disable_irq()`关中断，避免跳转过程中触发中断导致异常。
6. **VTOR + MSP + 复位向量：** `SCB->VTOR = app_addr`设置向量表偏移，`__set_MSP(sp)`设置主栈指针，从`app_addr + 4`读取Reset_Handler地址，函数指针调用跳转。

### Q20: 为什么用卡尔曼滤波器？不用会怎样？

**问题：** MPU6050的加速度计和陀螺仪各有优缺点：
- **加速度计：** 短期噪声大（振动、运动加速度干扰），但长期准确——测量的是重力方向，不会漂移。
- **陀螺仪：** 短期响应快、精度高，但长期会漂移——角速度积分会累积误差。

**如果只用加速度计：** 静止时Roll/Pitch会因为振动而抖动，水平仪气泡跳来跳去，用户体验差。

**如果只用陀螺仪积分：** Roll/Pitch会逐渐漂移，几分钟后角度就不准了。

**卡尔曼滤波融合的效果：** 短期信任陀螺仪（响应快），长期信任加速度计（不漂移）。零偏估计能自动补偿陀螺仪的漂移——即使陀螺仪有固定的偏移量，滤波器也能在几秒内估计出来并补偿掉。

**实现细节：**
- 2状态（角度 + 陀螺仪零偏）的一维卡尔曼滤波器，对Roll和Pitch各一个实例。
- `Kalman_GetAngle(&K_Roll, roll_a, gx / 16.4f, Delta_t)`：第一个参数是加速度计计算的角度（观测值），第二个是陀螺仪角速度（控制输入），第三个是时间步长。
- 16.4是±2000dps量程下的灵敏度（LSB/dps）。

### Q21: 卡尔曼滤波的参数Q_angle、Q_bias、R_measure是怎么调的？

- `Q_angle = 0.001`：角度过程噪声，越小越信任陀螺仪积分（响应快但可能漂移）
- `Q_bias = 0.003`：零偏过程噪声，控制零偏估计的收敛速度
- `R_measure = 0.03`：加速度计测量噪声，越大越不信任加速度计（平滑但响应慢）

**调参过程：**
1. **静止测试：** 把手表放在桌面上，观察Roll/Pitch输出是否稳定。如果R_measure太小（太信任加速度计），输出会随振动抖动；如果太大，响应会很迟钝。
2. **动态测试：** 快速转动手腕，观察角度是否及时跟随。如果Q_angle太小（太信任陀螺仪），动态响应快但可能漂移；如果太大，会过度依赖加速度计导致延迟。
3. **零偏收敛：** 静止时观察bias估计值是否收敛到稳定值（陀螺仪的静态偏移），Q_bias控制收敛速度——太大会导致bias估计不稳定，太小会导致收敛太慢。
4. **最终平衡：** 手动试参，逐步调整到"静止不抖动、动态不延迟"的平衡点。

### Q22: 计步算法的原理？怎么过滤不同场景的误触发？

项目使用的是**双阈值 + 双间隔**方案（step_algo.c），不是简单的峰值检测：

**算法流程：**
1. 每次采样计算加速度幅值：`magnitude = sqrt(ax² + ay² + az²)`
2. 计算与上次幅值的差值：`Delta = abs(magnitude - last_magnitude)`
3. 判断条件（四个条件同时满足才算一步）：
   - `Delta > STEP_THRESHOLD_MIN (2000)`：排除微小抖动（拿笔写字、敲键盘等不会触发）
   - `Delta < STEP_THRESHOLD_MAX (8000)`：排除异常冲击（跑步时手臂大幅摆动、碰撞等）
   - `间隔 > STEP_INTERVAL_MIN (200ms)`：排除高频噪声（手臂振动、骑车颠簸等，正常跑步最快也要200ms一步）
   - `间隔 < STEP_INTERVAL_MAX (2000ms)`：排除慢悠悠的走（超过2秒不算步频，可能是无意识的晃动）

**为什么是双阈值而不是单阈值：**
- 只有下限：走路能检测到，但手抖、碰撞也会触发（误触发率高）
- 只有上限：能过滤异常冲击，但正常跑步的大幅摆臂也会被过滤掉（漏检率高）
- 双阈值：只接受"像正常步态"的加速度变化，走路（Delta~3000-5000）能检测到，手抖（Delta<2000）和碰撞（Delta>8000）都被过滤

**阈值的单位：** MPU6050配置为±16g量程，灵敏度2048 LSB/g。阈值2000对应约1g的变化量，8000对应约3.9g——正常走路加速度变化约0.5-1.5g，跑步约1.5-3g，超过4g基本是异常冲击。

### Q23: FreeRTOS任务是怎么划分的？为什么这样设计？

4个任务按功能划分：
- **StartTask（优先级4）：** 初始化所有模块（DataCenter、UI、Serial、Queue、Mutex），创建其他3个任务后`vTaskDelete(NULL)`自删除，释放512字的栈空间。
- **KeyProcessTask（优先级3）：** 阻塞在`xQueueReceive(KeyQueue)`上等待按键事件，收到后调用`UI_HandleKey`分发。用户交互响应最及时。
- **DisplayTask（优先级2）：** 50ms周期刷新OLED（20fps），用`vTaskDelayUntil`保证精确周期。比传感器优先级高，保证画面流畅。
- **MPU6050Task（优先级1）：** 传感器采样+卡尔曼滤波+计步+抬腕检测，周期性任务（5ms或100ms），优先级最低，延迟几十ms可接受。

**优先级设计原则：** 用户交互(3) > 显示(2) > 传感器(1)。按键通过队列传递，阻塞等待不占CPU；显示需要流畅；传感器是后台数据采集。

---

### Q24: 为什么按键任务栈只分配了90字（360字节）？

KeyProcessTask逻辑很轻量——阻塞在`xQueueReceive`上，收到按键后调用`UI_HandleKey`分发到各页面处理函数。函数调用链很浅（`KeyProcessTask` → `UI_HandleKey` → `Page_*_HandleKey`），没有大局部变量。通过`uxTaskGetStackHighWaterMark`和`configCHECK_FOR_STACK_OVERFLOW = 2`（检查栈顶20字节的pattern `0xA5A5A5A5`）监控，确认没有溢出。

### Q25: `vTaskDelayUntil`和`vTaskDelay`有什么区别？

- `vTaskDelay(n)`：从函数返回时刻开始延时n个tick，如果函数本身执行耗时不确定，会导致周期漂移。
- `vTaskDelayUntil(&xLastWakeTime, xFrequency)`：基于绝对时间点延时，保证固定周期。

我在DisplayTask中用了`vTaskDelayUntil`，确保50ms的精确刷新周期。如果用`vTaskDelay`，每次循环执行时间不同会导致帧率不稳定。

### Q26: 你的系统里有共享资源访问的问题吗？怎么处理的？

两个共享资源：
1. **OLED显示缓冲：** DisplayTask写入，用`OLEDMutex`互斥锁保护。`xSemaphoreTake(OLEDMutex, portMAX_DELAY)`包裹`UI_Refresh()` + `OLED_Update()`。
2. **传感器数据（Roll/Pitch/Yaw）：** MPU6050Task写入，UI页面读取，通过`DataCenter`模块封装，内部用`SensorMutex`保护。

注意：秒表的`StopWatch_Tick`在TIM2 ISR中调用，没有加锁，因为它只操作简单的`uint8_t`/`uint16_t`变量，且只有ISR一个写者，读写是原子的。

### Q27: 互斥锁和信号量有什么区别？为什么用互斥锁？

- **互斥锁（Mutex）：** 有优先级继承机制——如果低优先级任务持锁，高优先级任务等待时，低优先级任务会临时提升优先级，避免优先级反转。适合保护共享资源。
- **信号量（Semaphore）：** 没有优先级继承，适合用于同步（ISR通知任务），不适合保护共享资源。

项目中OLED和传感器数据是共享资源，用互斥锁保护。按键事件用队列传递，本质是ISR到任务的同步机制。

### Q28: 为什么Yaw没有用卡尔曼滤波？

因为Yaw（偏航角）需要磁力计提供绝对参考。加速度计可以测量Roll/Pitch相对重力方向的倾角，但无法区分水平面内的旋转和移动。项目没有磁力计，Yaw只能靠陀螺仪角速度积分（`Yaw += (gz / 16.4f) * Delta_t`），会有累积漂移。这种情况下卡尔曼滤波没有可观测量来修正，所以直接用积分。实际使用中Yaw仅用于姿态参考，不作为精确航向。

### Q29: ESP8266的HTTP响应是怎么解析的？

`ESP8266_ReceiveHTTP`使用三状态状态机解析ESP8266的`+IPD,<len>:<data>`协议：
1. **WAIT_PLUS：** 逐字节读取串口数据，用`strstr`匹配`+IPD`标识
2. **READ_IPD_LEN：** 解析逗号后的数据长度字段
3. **READ_IPD_DATA：** 逐字节读取数据，通过计数`\r\n\r\n`（4个CRLF字符）跳过HTTP头部，只将body写入输出缓冲区。检测到"CLOSED"表示连接结束。

**为什么用状态机而不是直接解析：** HTTP响应可能分多次`+IPD`上报，且数据中可能包含二进制内容（不能用字符串函数），必须逐字节处理。

---

### Q30: W25Q128在OTA中扮演什么角色？

W25Q128（16MB SPI NOR Flash）作为固件**暂存区**（staging area）：
- ESP8266下载速度不确定，如果边下载边写片内Flash，中途断开会导致Flash处于不一致状态
- 片内Flash擦除粒度大（最小16KB），下载到一半失败代价大
- W25Q128的4KB扇区擦除快，适合频繁写入
- 下载和升级解耦，可以先下载完、校验通过，再决定是否升级

### Q31: 如果让你优化这个项目，你会怎么做？

1. **性能：** MPU6050数据读取改为burst read（从0x3B连续读14字节），一次I2C事务代替24次，减少CPU等待。
2. **功耗：** 使能Tickless Idle（代码已移植但未启用），实现STM32 Stop模式 + RTC闹钟唤醒，深度睡眠功耗可降至~20μA。
3. **通信：** ESP8266 AT指令驱动改为中断驱动 + 环形缓冲区，当前轮询方式浪费CPU。
4. **OTA：** 增加断点续传和固件加密签名，防止降级和中间人攻击。
5. **算法：** 计步算法增加频率分析（FFT或自相关），提高复杂运动场景下的准确率。

## 四、嵌入式基础知识

### Q32: FreeRTOS的抢占式调度是怎么工作的？

FreeRTOS使用抢占式调度 + 同优先级时间片轮转：
- **抢占式：** 高优先级任务就绪时立即抢占低优先级任务。
- **时间片轮转：** 同优先级任务轮流执行，每个任务运行一个tick（1ms）。
- **就绪列表：** 用`configUSE_PORT_OPTIMISED_TASK_SELECTION = 1`，ARM的CLZ指令可以在O(1)时间找到最高优先级的就绪任务。
- **阻塞/挂起：** 任务调用`vTaskDelay`、`xQueueReceive`等会进入阻塞态，让出CPU。

### Q33: 中断优先级和FreeRTOS优先级有什么关系？

两者是完全不同的概念：
- **FreeRTOS任务优先级：** 软件层面，数字越大优先级越高，范围0-7。
- **NVIC中断优先级：** 硬件层面，数字越小优先级越高，STM32F407有4位（16级）。
- **关系：** `configLIBRARY_MAX_SYSCALL_INTERRUPT_PRIORITY = 5`，意味着优先级0-4的中断**不能调用FreeRTOS API**，优先级5-15的中断可以。这是为了保护FreeRTOS内核的关键段不被更高优先级的中断打断。

### Q34: FreeRTOS的堆管理你用的哪种？为什么？

用的`heap_4`，支持合并相邻空闲块，适合频繁分配/释放不同大小内存的场景。项目配置了32KB堆空间。Idle Task和Timer Task用静态分配（`configSUPPORT_STATIC_ALLOCATION = 1`），不占动态堆。

### Q35: 你是怎么检测栈溢出的？

配置了`configCHECK_FOR_STACK_OVERFLOW = 2`，FreeRTOS在上下文切换时检查栈顶20字节是否被破坏（pattern `0xA5A5A5A5`）。如果检测到溢出，`vApplicationStackOverflowHook`会打印任务名并进入死循环。另外还有运行时`uxTaskGetStackHighWaterMark`监控栈剩余量。

### Q36: `__WFI()`的原理是什么？和`__WFE()`有什么区别？

- `__WFI()`（Wait For Interrupt）：使CPU进入低功耗状态，直到发生任意中断才唤醒。
- `__WFE()`（Wait For Event）：等待事件信号，可以用SEV指令唤醒，适合多核场景。

在单核MCU上，WFI更常用，因为SysTick中断会自然唤醒CPU。WFI只暂停CPU内核，外设和NVIC仍正常工作。

### Q37: 软件I2C和硬件I2C有什么区别？你各用在什么地方？

- **软件I2C（GPIO模拟）：** 时序完全可控，适合对时序要求严格的设备。项目中MPU6050和OLED用软件I2C（PB12/PB13和PB14/PB15）。
- **硬件I2C：** 有ACK/NACK自动处理，代码更简洁，但某些MCU的硬件I2C有已知的总线锁死问题。项目中AT24C02用硬件I2C（PB6/PB7，I2C1）。

### Q38: SPI和I2C有什么区别？你项目中分别用在哪些地方？

| 特性 | SPI | I2C |
|------|-----|-----|
| 线数 | 4线（SCK/MISO/MOSI/CS） | 2线（SCL/SDA） |
| 速度 | 可达几十MHz | 标准100kHz/400kHz/3.4MHz |
| 寻址 | CS片选，一对一 | 7位地址，多设备共享 |
| 全双工 | 是 | 否（半双工） |

项目中：
- **SPI：** W25Q128 Flash（SPI1，PA5/PA6/PA7，CS=PC3）
- **I2C：** MPU6050（软件I2C）、OLED（软件I2C）、AT24C02（硬件I2C1）

### Q39: DMA是什么？你在项目中用到了吗？

DMA（Direct Memory Access）允许外设和内存之间直接传输数据，不需要CPU逐字节搬运。项目中：
- **USART3的DMA：** ESP8266的串口通信理论上可以用DMA，但当前实现是中断驱动+环形缓冲区。
- **未充分利用：** MPU6050的数据读取是逐寄存器读取，可以优化为burst read + DMA。

### Q40: NVIC中断控制器的作用是什么？

NVIC（Nested Vectored Interrupt Controller）是ARM Cortex-M的中断管理核心：
- **中断使能/禁用：** 可以单独使能或禁用每个中断源。
- **优先级管理：** 4位优先级（16级），支持抢占和子优先级。
- **向量表：** 中断发生时自动跳转到向量表中对应的处理函数。
- **尾链优化：** 连续中断时省去入栈/出栈开销，降低延迟。

### Q41: STM32的时钟系统是怎么配置的？

STM32F407的时钟树：
- **HSE：** 8MHz外部晶振
- **PLL：** 8MHz × 21 / 2 = 168MHz（系统时钟SYSCLK）
- **AHB：** 168MHz（HCLK，CPU和DMA）
- **APB1：** 42MHz（低速外设，USART2/3、TIM2-7等）
- **APB2：** 84MHz（高速外设，USART1、SPI1、ADC等）
- **USB：** 48MHz（需要精确的48MHz，由PLL的Q分频得到）

### Q42: Cortex-M4的中断响应过程是什么？

1. **入栈：** 硬件自动将PC、xPSR、R0-R3、R12、LR压入当前栈。
2. **取向量：** 从向量表中读取对应ISR的地址。
3. **选择栈指针：** 根据EXC_RETURN值选择MSP或PSP。
4. **更新寄存器：** 更新NVIC的活跃中断标志。

延迟：从中断触发到ISR第一条指令执行，最快12个时钟周期（Cortex-M4）。

### Q43: volatile关键字的作用是什么？你在哪些场景用了？

`volatile`告诉编译器：该变量可能在编译器不可见的情况下被修改（如ISR、硬件寄存器、DMA），禁止编译器优化掉对该变量的读写。

项目中的使用场景：
- `volatile TickType_t LastActionTime`：在ISR（按键中断）中更新，在DisplayTask中读取。
- `volatile uint8_t IsScreenOn`：在多任务间共享的状态标志。
- 外设寄存器：如`GPIOA->IDR`读取按键状态。

### Q44: 什么是临界区？FreeRTOS中怎么实现？

临界区是一段不能被中断打断的代码。FreeRTOS中：
- **taskENTER_CRITICAL()：** 关闭中断（修改BASEPRI寄存器），保护短小的临界区。
- **taskEXIT_CRITICAL()：** 恢复中断。
- **互斥锁/信号量：** 保护较长的共享资源访问，允许任务阻塞等待。

项目中OLED操作用互斥锁而非临界区，因为OLED刷新耗时较长（几十ms），关中断太久会影响实时性。

### Q45: 什么是优先级反转？怎么解决？

优先级反转：低优先级任务持锁，高优先级任务等待该锁，此时中优先级任务抢占低优先级任务运行，导致高优先级任务间接被中优先级任务阻塞。

解决方案：
- **优先级继承（Priority Inheritance）：** FreeRTOS的Mutex支持。高优先级任务等待时，持锁任务临时提升到高优先级。
- **优先级天花板（Priority Ceiling）：** 任务持锁时直接提升到最高优先级。

项目中用Mutex而非二值信号量，就是因为Mutex有优先级继承机制。

### Q46: 什么是死锁？怎么避免？

死锁：两个任务互相等待对方持有的锁，导致两者都无法继续。

避免方法：
- **固定加锁顺序：** 所有任务按照相同的顺序获取多个锁。
- **超时机制：** `xSemaphoreTake(mutex, timeout)` 设置超时，超时后释放已持有的锁。
- **项目中的做法：** 系统只有两个Mutex（OLEDMutex和SensorMutex），且不存在一个任务同时需要两个锁的场景，所以不会死锁。

### Q47: UART串口通信的基本参数有哪些？

- **波特率（Baud Rate）：** 每秒传输的符号数，项目中ESP8266用115200bps。
- **数据位（Data Bits）：** 每帧的数据位数，通常8位。
- **停止位（Stop Bits）：** 帧结束标志，通常1位。
- **校验位（Parity）：** 奇偶校验，通常无校验。
- **流控（Flow Control）：** 硬件流控（RTS/CTS）或软件流控（XON/XOFF），项目中未使用。

### Q48: 中断服务程序（ISR）中需要注意什么？

1. **尽量短：** ISR中只做必要的操作（读数据、发信号），耗时操作交给任务处理。
2. **不能阻塞：** 不能调用`vTaskDelay`、`xSemaphoreTake`等可能阻塞的API。
3. **FromISR后缀：** FreeRTOS提供了`xQueueSendFromISR`、`xSemaphoreGiveFromISR`等专用API。
4. **中断优先级：** 调用FreeRTOS API的中断优先级必须≥`configLIBRARY_MAX_SYSCALL_INTERRUPT_PRIORITY`（项目中为5）。
5. **清除中断标志：** 必须手动清除中断标志位，否则会反复进入ISR。

### Q49: 什么是栈（Stack）和堆（Heap）？

- **栈（Stack）：** 由编译器自动管理，用于函数调用时保存局部变量、返回地址等。空间有限（任务栈通常几百到几千字节），分配/释放速度极快（移动栈指针即可）。
- **堆（Heap）：** 由程序员手动管理（`malloc`/`free`），用于动态分配内存。空间较大（项目中FreeRTOS堆32KB），可能产生碎片。

FreeRTOS中每个任务有独立的栈，堆是所有任务共享的。

### Q50: 什么是看门狗（Watchdog）？你项目中用了吗？

看门狗是一个硬件定时器，如果程序在规定时间内没有"喂狗"（重置计数器），看门狗会触发系统复位。用于防止程序跑飞或死循环。

项目中没有使用看门狗，因为FreeRTOS的任务调度本身就是一种"心跳"机制——如果某个任务卡死，`uxTaskGetStackHighWaterMark`和任务状态监控可以检测到。

### Q51: Flash和EEPROM有什么区别？

| 特性 | Flash | EEPROM |
|------|-------|--------|
| 擦除粒度 | 扇区（16KB-128KB） | 字节 |
| 写入速度 | 较快（编程） | 较慢（5ms写周期） |
| 擦写寿命 | 1万-10万次 | 100万次 |
| 容量 | 大（几百KB-几MB） | 小（几百字节-几KB） |
| 成本 | 低 | 高 |

项目中：Flash存储固件代码，EEPROM存储频繁更新的BootFlags（32字节）。

## 五、通信协议与网络

### Q52: HTTP和HTTPS有什么区别？

- **HTTP：** 明文传输，数据不加密。
- **HTTPS：** HTTP + TLS/SSL加密，数据加密传输，需要证书验证。

项目中ESP8266使用的是HTTP（明文），因为嵌入式设备资源有限，TLS握手和加解密开销较大。实际产品中应使用HTTPS或MQTT+TLS。

### Q53: TCP和UDP有什么区别？项目中分别用在哪里？

| 特性 | TCP | UDP |
|------|-----|-----|
| 连接 | 面向连接（三次握手） | 无连接 |
| 可靠性 | 可靠（重传、排序） | 不可靠 |
| 速度 | 较慢 | 较快 |
| 适用 | 文件传输、HTTP | 实时音视频、游戏 |

项目中：
- **TCP：** ESP8266连接HTTP服务器下载固件（OTA）
- **UDP：** xiaozhi-esp32项目中用于音频流传输（低延迟需求）

### Q54: MQTT协议是什么？有什么特点？

MQTT（Message Queuing Telemetry Transport）是轻量级的消息队列协议：
- **发布/订阅模式：** 客户端不直接通信，通过Broker中转。
- **主题（Topic）：** 消息按主题分类，支持通配符订阅。
- **QoS：** 支持三种服务质量等级（0-最多一次，1-至少一次，2-恰好一次）。
- **遗嘱消息：** 客户端异常断开时，Broker自动发布预设消息。

xiaozhi-esp32项目中用MQTT做设备管理通道（远程控制、OTA、资产同步）。

### Q55: WebSocket是什么？和HTTP有什么区别？

WebSocket是全双工通信协议：
- **HTTP：** 请求-响应模式，客户端发起，服务端响应，无状态。
- **WebSocket：** 建立连接后，双方可以随时主动发送数据，适合实时通信。

xiaozhi-esp32项目中用WebSocket传输音频流，因为语音交互需要低延迟的双向通信。

### Q56: 项目中遇到过哪些Bug？怎么解决的？

**智能手表项目：**

## 八、C/C++基础

### Q64: C语言中static关键字的作用？

1. **局部变量：** 生命周期变为整个程序运行期间，但作用域不变（只在函数内可见）。
2. **全局变量/函数：** 限制为文件作用域（internal linkage），其他文件不能访问。
3. **项目中的使用：** 任务函数声明为static（如`static void DisplayTask(void *pvParameters)`），限制在task_manager.c内可见。卡尔曼滤波器实例`static Kalman_t K_Roll, K_Pitch`限制在task_sensor.c内。

### Q65: C语言中指针和数组的区别？

- **数组：** 连续的内存块，`sizeof(arr)`返回整个数组大小。
- **指针：** 存储地址的变量，`sizeof(ptr)`返回指针本身大小（4字节 on 32位系统）。
- **关键区别：** 数组名在大多数表达式中会退化为指向首元素的指针，但`sizeof`和`&`操作除外。

### Q66: 什么是内存对齐？为什么重要？

内存对齐要求数据存储在特定边界的地址上（如4字节数据存在4的倍数地址）。原因：
- **硬件要求：** ARM Cortex-M要求对齐访问，非对齐访问会触发HardFault（或性能下降）。
- **性能：** 对齐访问可以在一个时钟周期内完成，非对齐可能需要多次访问。

### Q67: C++中struct和class的区别？

- **默认访问权限：** struct默认public，class默认private。
- **默认继承方式：** struct默认public继承，class默认private继承。
- **其他完全相同：** 都可以有成员函数、虚函数、模板等。

项目中C代码用struct定义数据结构（如`BootFlags_t`），C++代码用class定义有行为的对象（如`Application`、`Board`）。

### Q68: 什么是RAII？有什么好处？

RAII（Resource Acquisition Is Initialization）：资源获取即初始化。在构造函数中获取资源（内存、文件句柄、锁等），在析构函数中释放资源。

好处：
- **防止资源泄漏：** 即使发生异常，析构函数也会被调用。
- **代码简洁：** 不需要手动`free`/`close`。

xiaozhi-esp32项目中OPUS编码器的C++封装就是RAII的典型应用。

### Q69: 什么是智能指针？嵌入式中能用吗？

智能指针是RAII包装原始指针的类：
- `std::unique_ptr`：独占所有权，不能拷贝。
- `std::shared_ptr`：引用计数，共享所有权。
- `std::weak_ptr`：不增加引用计数的观察者。

嵌入式中可以用`std::unique_ptr`（开销很小），但`std::shared_ptr`有额外的引用计数开销和动态内存分配，不适合资源受限的MCU。xiaozhi-esp32（ESP32-S3，有PSRAM）中可以使用，STM32项目中一般不用。

## 九、操作系统基础

### Q70: 什么是RTOS？和Linux有什么区别？

RTOS（实时操作系统）的核心特征是**确定性**——任务能在可预测的时间内得到响应。

| 特性 | FreeRTOS | Linux |
|------|----------|-------|
| 内核 | 微内核，几十KB | 宏内核，几MB |
| 调度 | 抢占式，确定性延迟 | CFS公平调度 |
| 内存保护 | 无（默认无MPU） | 有（MMU虚拟内存） |
| 实时性 | 硬实时（μs级响应） | 软实时（ms级） |
| 适用 | MCU（Cortex-M） | MPU（Cortex-A） |

### Q71: 什么是任务（Task）、线程（Thread）、进程（Process）？

- **进程：** 拥有独立地址空间的程序实例。Linux中进程间内存隔离。
- **线程：** 进程内的执行单元，共享进程的地址空间。
- **任务：** FreeRTOS中的"任务"等价于线程（共享同一地址空间），但FreeRTOS没有进程概念。

### Q72: 什么是上下文切换（Context Switch）？

上下文切换是保存当前任务的CPU寄存器状态、恢复下一个任务的CPU寄存器状态的过程。Cortex-M4硬件自动保存部分寄存器（入栈），FreeRTOS内核保存剩余寄存器（R4-R11）。

开销：几十个时钟周期（~1μs @168MHz），但频繁切换会累积影响性能。

### Q73: 什么是信号量、互斥锁、事件组？有什么区别？

| 机制 | 用途 | 特点 |
|------|------|------|
| 二值信号量 | 同步（ISR→任务） | 无优先级继承 |
| 计数信号量 | 资源计数 | 如缓冲区空闲槽位数 |
| 互斥锁 | 保护共享资源 | 有优先级继承 |
| 事件组 | 多条件同步 | 支持AND/OR等待 |

## 十、项目设计与架构

### Q74: 你项目中的数据流是怎么走的？

**智能手表数据流：**
```
MPU6050 (I2C) → MPU6050Task → 卡尔曼滤波 → DataCenter(SensorMutex) → UI页面读取 → OLED显示
按键 (GPIO) → TIM2 ISR → KeyQueue → KeyProcessTask → UI_HandleKey → 页面切换
```

**xiaozhi-esp32数据流：**
```
麦克风 (I2S) → AudioProcessor(AFE) → OPUS编码 → WebSocket/MQTT → 云端ASR → LLM → TTS → OPUS解码 → 扬声器 (I2S)
```

### Q75: 如果让你从零设计一个智能手表，你会怎么做？

1. **硬件选型：** 选择低功耗MCU（如STM32L4系列或nRF52840），集成BLE。
2. **RTOS选型：** FreeRTOS或Zephyr（更现代，驱动框架更好）。
3. **驱动层：** 统一的外设抽象层（HAL），方便硬件迭代。
4. **中间件：** 传感器融合、蓝牙协议栈、文件系统。
5. **应用层：** 页面管理器 + 功能模块，解耦UI和业务逻辑。
6. **OTA：** 从一开始就设计好A/B分区布局。
7. **功耗：** Tickless Idle + 动态频率调整 + 外设电源管理。

### Q76: 项目中你最得意的设计是什么？

**智能手表的OTA方案。** 因为涉及Bootloader、App、外部Flash、EEPROM四个组件的协调，任何一个环节出错都可能导致设备变砖。通过A/B分区保证升级失败不影响当前系统，双重CRC校验保证数据完整性，BootFlags的magic字段防止读取到随机数据，栈指针校验防止跳转到损坏的App。整个方案设计考虑了多种异常场景，确保不会变砖。

### Q77: 你在项目中做了哪些取舍（Trade-off）？

1. **MPU6050量程选择：** ±2000dps（陀螺仪）和±16g（加速度计）——牺牲精度换取动态范围，适合穿戴场景。
2. **OLED刷新率：** 20fps而非更高——在功耗和流畅度之间取平衡。
3. **ESP8266轮询 vs 中断驱动：** 选择了实现简单的轮询方式，但CPU利用率不是最优。
4. **软件I2C vs 硬件I2C：** MPU6050用软件I2C保证时序可控，AT24C02用硬件I2C简化代码。
