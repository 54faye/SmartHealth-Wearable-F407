# 智能手表项目 - 面试问答手册

---

## 一、项目概述

### Q1: 请简单介绍一下你的智能手表项目

基于 STM32F407VE + FreeRTOS 实现的多功能智能手表，独立完成从 BSP 驱动、RTOS 任务架构到上层应用的全流程开发。硬件上集成 MPU6050 六轴传感器、0.96 寸 OLED、W25Q128 外部 Flash、AT24C02 EEPROM 和 ESP8266 WiFi 模块。功能上实现了抬腕亮屏、卡尔曼滤波传感器融合、计步、水平仪、小游戏等，并设计了完整的 A/B 分区 OTA 升级方案和多级低功耗策略。

### Q2: 项目的整体软件架构是怎样的？

分为四层：
- **BSP 层：** 各外设驱动（MPU6050、OLED、W25Q128、AT24C02、ESP8266、RTC、按键、LED）
- **中间件层：** 卡尔曼滤波、计步算法、调试串口
- **RTOS 任务层：** 4 个 FreeRTOS 任务（StartTask 初始化后自删除、KeyProcessTask 按键处理、DisplayTask 显示刷新、MPU6050Task 传感器采集）
- **应用层：** UI 页面管理器 + 10 个功能页面（时钟、菜单、秒表、LED、MPU6050 数据、游戏、表情、水平仪、设置、计步）

### Q3: 你用了哪些 FreeRTOS 的机制？分别用来解决什么问题？

| 机制 | 用途 |
|------|------|
| 任务（Task） | 4 个独立任务，按优先级调度 |
| 队列（Queue） | 按键事件从 ISR 到 KeyProcessTask 的传递 |
| 互斥锁（Mutex） | 保护 OLED 显示缓冲和共享传感器数据 |
| Idle Hook | 调用 WFI 实现 CPU 低功耗 |
| 静态分配 | Idle Task 和 Timer Task 使用静态 TCB/Stack |
| `vTaskDelayUntil` | DisplayTask 精确周期调度 |

## 二、FreeRTOS 相关

### Q4: 你的任务优先级是怎么设计的？为什么这样排？

KeyProcessTask(3) > DisplayTask(2) > MPU6050Task(1)，StartTask(4) 初始化后自删除，Timer Task(7) 最高。
- **按键最高（3）：** 用户交互响应要最及时，按键事件通过队列传递，阻塞等待不占 CPU。
- **显示次之（2）：** 20fps 刷新，比传感器优先级高，保证画面流畅。
- **传感器最低（1）：** 传感器采样是周期性的，延迟几十毫秒可接受。

### Q5: 为什么按键任务栈只分配了 90 字（360 字节），够用吗？

KeyProcessTask 的逻辑很轻量——阻塞在 `xQueueReceive` 上，收到按键后调用 `UI_HandleKey` 分发到各页面的按键处理函数。函数调用链很浅，没有大局部变量，90 字完全够用。实际通过 `uxTaskGetStackHighWaterMark` 和 `configCHECK_FOR_STACK_OVERFLOW = 2` 监控，没有溢出。

### Q6: `vTaskDelayUntil` 和 `vTaskDelay` 有什么区别？你在哪里用了？

`vTaskDelay(n)` 是从函数返回时刻开始延时 n 个 tick，如果函数本身执行耗时不确定，会导致周期漂移。`vTaskDelayUntil(&xLastWakeTime, xFrequency)` 是基于绝对时间点延时，保证固定周期。我在 DisplayTask 中用了 `vTaskDelayUntil`，确保 50ms 的精确刷新周期（20fps）。

### Q7: 你的系统里有共享资源访问的问题吗？怎么处理的？

有两个共享资源：
1. **OLED 显示缓冲：** DisplayTask 写入，可能多个页面函数访问。用 `OLEDMutex` 保护，`xSemaphoreTake` / `xSemaphoreGive` 包裹所有 OLED 操作。
2. **传感器数据（Roll/Pitch/Yaw）：** MPU6050Task 写入，UI 页面读取。通过 `DataCenter` 模块封装，内部用 `SensorMutex` 保护。

注意：秒表的 `StopWatch_Tick` 在 TIM2 ISR 中调用，没有加锁，因为它只操作简单的 `uint8_t`/`uint16_t` 变量，且只有 ISR 一个写者，读写是原子的。

### Q8: 为什么用互斥锁而不是信号量？什么时候用信号量？

互斥锁有优先级继承机制——如果低优先级任务持锁，高优先级任务等待时，低优先级任务会临时提升优先级，避免优先级反转。纯二值信号量没有这个机制。信号量适合用于同步（ISR 通知任务），不适合保护共享资源。项目中按键用的是队列，本质也是 ISR 到任务的同步机制。

### Q9: FreeRTOS 的堆管理你用的哪种？为什么？

用的 `heap_4`，支持合并相邻空闲块，适合频繁分配/释放不同大小内存的场景。项目配置了 32KB 堆空间。Idle Task 和 Timer Task 用静态分配（`configSUPPORT_STATIC_ALLOCATION = 1`），不占动态堆。

### Q10: 你是怎么检测栈溢出的？

配置了 `configCHECK_FOR_STACK_OVERFLOW = 2`，FreeRTOS 在上下文切换时检查栈顶 20 字节（`STACK_WARNING_THRESHOLD`）是否被破坏（pattern `0xA5A5A5A5`）。如果检测到溢出，`vApplicationStackOverflowHook` 会打印任务名并进入死循环。另外还有运行时 `uxTaskGetStackHighWaterMark` 监控栈剩余量。

## 三、低功耗设计

### Q11: 你做了哪些低功耗优化？效果如何？

四级低功耗策略：
1. **CPU 级：** Idle Hook 中调用 `__WFI()`，无任务时 CPU 进入 Sleep 模式，等待中断唤醒。
2. **外设级：** OLED 3 分钟无操作自动熄屏，熄屏后 DisplayTask 切换为 500ms 轮询（降低 10 倍刷新开销）。
3. **传感器级：** 自适应采样率——查看姿态/水平仪页面时 200Hz，其余场景降至 10Hz，功耗降低约 95%。
4. **交互级：** 抬腕检测唤醒，无需按键即可点亮屏幕。

### Q12: `__WFI()` 的原理是什么？和 `__WFE()` 有什么区别？

`__WFI()`（Wait For Interrupt）使 CPU 进入低功耗状态，直到发生任意中断才唤醒。`__WFE()`（Wait For Event）等待事件信号，可以用 SEV 指令唤醒，适合多核场景。在单核 MCU 上，WFI 更常用，因为 SysTick 中断会自然唤醒 CPU。WFI 只暂停 CPU 内核，外设和 NVIC 仍正常工作。

### Q13: Tickless Idle 是什么？你实现了吗？

正常情况下 FreeRTOS 每 1ms 产生一次 SysTick 中断，即使系统完全空闲也会频繁唤醒 CPU。Tickless Idle 的原理是：当 Idle Task 运行且下一个定时事件还很远时，动态关闭 SysTick，让 CPU 进入深度睡眠，醒来后补上丢失的 tick。

项目中 FreeRTOS port 层的 `vPortSuppressTicksAndSleep()` 函数已经移植好，`app_config.h` 有 `ENABLE_TICKLESS_IDLE = 1` 的标志，但 `FreeRTOSConfig.h` 中 `configUSE_TICKLESS_IDLE` 未使能，所以代码已就绪但未正式启用。

### Q14: 抬腕检测的算法是怎么实现的？如何避免误触发？

- **原理：** 用 MPU6050 的 Roll/Pitch 角度判断手腕姿态，当 Roll 在 ±45°、Pitch 在 -45°~+10° 范围内时认为是"抬手看表"姿势。
- **防误触：** 使用**边沿触发**而非电平触发——只有从"不符合"到"符合"的状态跳变才触发亮屏，持续保持抬手姿势不会重复触发。另外，屏幕亮着时实时更新 `last_posture_ok` 状态，确保放下手再抬起时能正确触发。

### Q15: 自适应采样率是怎么实现的？切换时会不会丢数据？

在 MPU6050Task 的每次循环末尾判断：如果屏幕亮着且当前页面是 MPU6050 或水平仪，采样周期设为 5ms（200Hz），否则设为 100ms（10Hz）。切换发生在当前循环结束后，不会中断正在进行的采样。卡尔曼滤波器的 `dt` 参数也同步调整，保证融合算法的正确性。

## 四、传感器与算法

### Q16: 卡尔曼滤波器的原理？你的实现有几个状态变量？

实现的是 2 状态（角度 + 陀螺仪零偏）的一维卡尔曼滤波器。
- **状态向量：** `[angle, bias]`
- **预测步：** 用陀螺仪角速度减去估计的零偏，积分预测角度，同时更新误差协方差矩阵。
- **更新步：** 用加速度计计算的倾角作为观测量，计算卡尔曼增益，修正角度和零偏估计。
- **核心价值：** 加速度计短期噪声大但长期准确（不会漂移），陀螺仪短期响应快但长期会漂移。卡尔曼滤波融合两者优点，零偏估计能自动补偿陀螺仪漂移。

### Q17: 卡尔曼滤波的参数 Q_angle、Q_bias、R_measure 是怎么调的？

这三个参数分别控制：
- `Q_angle = 0.001`：角度过程噪声，越小越信任陀螺仪积分（响应快但可能漂移）
- `Q_bias = 0.003`：零偏过程噪声，控制零偏估计的收敛速度
- `R_measure = 0.03`：加速度计测量噪声，越大越不信任加速度计（平滑但响应慢）

调参方法：先从默认值开始，观察静止时输出是否稳定（R_measure 太小会抖动），再看动态响应是否及时（Q_angle 太大会延迟）。实际是手动试参，逐步调整到动静态都可接受的平衡点。

### Q18: 为什么 Yaw 没有用卡尔曼滤波？

因为 Yaw（偏航角）需要磁力计提供绝对参考。加速度计可以测量 Roll/Pitch 相对重力方向的倾角，但无法区分水平面内的旋转和移动。项目没有磁力计，Yaw 只能靠陀螺仪角速度积分，会有累积漂移。这种情况下卡尔曼滤波没有可观测量来修正，所以直接用积分。实际使用中 Yaw 仅用于姿态参考，不作为精确航向。

### Q19: 计步算法的原理？两种实现有什么区别？

**方案一（step_algo.c）—— 双阈值 + 双间隔：**
- 计算加速度幅值 `sqrt(ax² + ay² + az²)`，与上一次的差值 Delta
- 检测条件：`2000 < Delta < 8000`（排除微小抖动和异常冲击）且 `200ms < 间隔 < 2000ms`（匹配正常步频）
- 优点：抗干扰能力强，双阈值过滤噪声

**方案二（pedometer.c）—— 峰谷状态机：**
- 以 1g 为基线，加速度超过 1g+2000 记为峰值，回落到 1g 以下记为谷值，峰-谷间隔 > 300ms 算一步
- 优点：逻辑直观，容易理解

项目实际使用的是方案一（step_algo.c），因为它对异常加速度（如手抖、碰撞）的过滤更好。

### Q20: MPU6050 的配置为什么选择 ±2000dps 和 ±16g？

- **±2000dps（陀螺仪）：** 手腕运动角速度较大，±250dps 容易饱和。±2000dps 灵敏度为 16.4 LSB/dps，虽然精度略低，但动态范围大，适合穿戴场景。
- **±16g（加速度计）：** 日常使用 ±2g 够用，但运动中可能有冲击。±16g 灵敏度 2048 LSB/g，留足余量。计步算法的阈值参数（2000、8000）就是在这个量程下标定的。

### Q21: MPU6050 的数据读取方式有什么可以优化的地方？

当前实现是逐寄存器读取，每个轴 2 字节需要 2 次 I2C 事务，6 轴共 24 次事务。可以优化为：
1. **Burst Read：** 从 `ACCEL_XOUT_H`（0x3B）起始连续读 14 字节（6 轴 + 温度），一次 I2C 事务搞定，减少 23 次通信开销。
2. **使用硬件 I2C：** 当前用的是软件 I2C（GPIO 模拟），切换到 STM32 的硬件 I2C 可以用 DMA 传输，CPU 完全不用等待。

## 五、OTA 升级

### Q22: 你的 OTA 方案的整体架构是什么？

A/B 双分区 + 外部暂存的经典方案：
1. **分区布局：** Bootloader（0x08000000, 64KB）+ App_A（0x08010000, 192KB）+ App_B（0x08040000, 256KB）
2. **下载阶段：** App 通过 ESP8266 WiFi 从 HTTP 服务器下载固件到 W25Q128 外部 Flash
3. **校验阶段：** 用 STM32 硬件 CRC32 外设计算固件校验和，与服务器下发的 CRC 比对
4. **标记阶段：** 校验通过后，将 BootFlags（magic、目标分区、CRC、长度、版本）写入 AT24C02 EEPROM
5. **升级阶段：** 系统复位，Bootloader 读取 BootFlags，从 W25Q128 搬运固件到目标分区，搬运前后双重 CRC 校验
6. **启动阶段：** 校验通过后设置 active_partition，跳转 App

### Q23: 为什么用 A/B 分区而不是单分区 + 备份？

A/B 分区的核心优势是**升级失败不会变砖**：
- 单分区方案：升级过程中断电 → 固件写了一半 → 新固件不完整，旧固件已被擦除 → 变砖
- A/B 方案：新固件写入非活动分区，当前运行的分区完全不受影响。即使升级中断，Bootloader 检测到 `ota_pending` 不为 1 或 CRC 不匹配，直接跳转到当前活动分区，系统正常运行。

### Q24: BootFlags 为什么存在 EEPROM 而不是 Flash？

因为 BootFlags 需要在 OTA 流程中被频繁更新（写入 magic → 写入 CRC → 升级后清除 pending），而且需要在 Bootloader 和 App 之间共享。
- **Flash 的问题：** STM32F407 的 Flash 擦除以扇区为单位（最小 16KB），写入 BootFlags 32 字节需要先擦整个扇区，效率低且有磨损。
- **EEPROM 优势：** AT24C02 支持字节级写入，100 万次写入寿命，不需要擦除操作，读写速度快（I2C 400kHz），适合存储频繁更新的小数据。

### Q25: CRC32 校验具体怎么做的？为什么用硬件 CRC？

从 W25Q128 按 256 字节分块读取固件，每块转为 32-bit 字送入 STM32F407 的硬件 CRC 单元（`RCC_AHB1Periph_CRC`），不足 4 字节的尾部用 0xFF 填充。最终 CRC 值与 BootFlags 中存储的 `fw_crc32` 比对。

用硬件 CRC 的原因：软件 CRC32 计算一个 ~192KB 的固件需要几十毫秒，而硬件 CRC 外设每个时钟周期处理一个 32-bit 字，168MHz 下 192KB 只需约 0.3ms，快两个数量级。

### Q26: Bootloader 跳转 App 前做了哪些安全检查？

1. **栈指针校验：** 读取 App 起始地址的第一个字（初始 SP），检查是否指向 SRAM 区域（`(SP & 0x2FF00000) == 0x20000000`）。如果 App 未烧写或损坏，这个值会是 0xFFFFFFFF，校验失败不会跳转。
2. **中断关闭：** 跳转前调用 `__disable_irq()` 关中断，防止跳转过程中触发中断导致异常。
3. **SysTick 关闭：** 避免 Bootloader 的 SysTick 干扰 App。
4. **VTOR 设置：** 将向量表偏移寄存器 `SCB->VTOR` 设置为 App 地址，确保 App 的中断向量表正确。
5. **MSP 设置：** 将主栈指针设为 App 的初始 SP。
6. **读取复位向量：** 从 `app_addr + 4` 读取 App 的 Reset_Handler 地址，函数指针调用跳转。

### Q27: 如果 OTA 下载过程中断网了怎么办？

不会影响当前系统运行。OTA 下载是 App 侧的操作，固件先写入 W25Q128 外部 Flash，不影响当前运行的 App 分区。断网后 `ESP8266_ReceiveHTTP` 超时返回，`OTA_DownloadFirmware` 返回错误码，不会设置 BootFlags。Bootloader 上电后不会看到 `ota_pending = 1`，直接跳转当前活动分区，系统正常运行。

### Q28: OTA_MAGIC 0x4F544131 是什么含义？

这是 ASCII 字符串 "OTA1" 的十六进制表示（O=0x4F, T=0x54, A=0x41, 1=0x31）。用作魔数（magic number）来验证 BootFlags 是否有效——AT24C02 上电后内容是随机的，如果 magic 不等于这个值，说明没有合法的 OTA 标记，Bootloader 直接跳转默认分区。

## 六、ESP8266 通信

### Q29: ESP8266 的驱动是怎么设计的？如何解析 HTTP 响应？

ESP8266 通过 AT 指令控制，USART3 通信。核心设计：
1. **AT 指令发送：** `esp_send_cmd(cmd, ack, timeout)` 发送指令并等待预期响应。
2. **等待响应：** `esp_wait_for(target, timeout)` 逐字节读取串口数据到缓冲区，用 `strstr` 匹配目标字符串。
3. **HTTP 解析状态机：** `ESP8266_ReceiveHTTP` 有三个状态——`WAIT_PLUS`（等待 `+IPD`）→ `READ_IPD_LEN`（解析数据长度）→ `READ_IPD_DATA`（读取数据）。通过计数 `\r\n\r\n`（4 个 CRLF 字符）跳过 HTTP 头部，只将 body 写入缓冲区。

### Q30: ESP8266 的 `+IPD` 协议是什么？为什么要用状态机解析？

ESP8266 在 TCP 模式下收到数据时，会以 `+IPD,<len>:<data>` 的格式上报。`<len>` 是数据长度，`<data>` 是实际内容。因为数据可能分多次 `+IPD` 上报，且 HTTP 响应包含头部和 body，需要状态机逐字节解析：
- 在 `READ_IPD_DATA` 状态中计数 CRLF，跳过 HTTP 头部
- 只将 body 部分写入输出缓冲区
- 检测到 "CLOSED" 表示连接结束

### Q31: 512 字节的 `esp_buf` 够用吗？如果 HTTP 响应超过 512 字节呢？

`esp_buf` 用于解析 AT 指令响应和 HTTP 头部，不是存储整个固件。固件下载时，`ESP8266_ReceiveHTTP` 每次 `+IPD` 返回一小块数据（通常几百字节），直接写入 W25Q128，不缓存在 `esp_buf` 中。所以 512 字节够用。

## 七、硬件驱动

### Q32: 软件 I2C 和硬件 I2C 你都用了，各用在什么地方？为什么这样选择？

- **软件 I2C（GPIO 模拟）：** MPU6050（PB12/PB13）和 OLED（PB14/PB15）。因为这两个设备共享 GPIO 端口 B，且 MPU6050 有时序要求严格的 DMP 读取，软件 I2C 时序完全可控。
- **硬件 I2C：** AT24C02（PB6/PB7，I2C1）。EEPROM 读写简单，硬件 I2C 有 ACK/NACK 自动处理，代码更简洁。

### Q33: MPU6050 的 DLPF（数字低通滤波器）配置为 6 是什么含义？

CONFIG 寄存器值 0x06 表示 DLPF 带宽约 5Hz，延迟约 18.6ms。这意味着加速度计和陀螺仪输出都经过 5Hz 低通滤波，滤除高频噪声。代价是引入约 19ms 延迟，对于穿戴设备的姿态检测可以接受。如果需要更快的响应（比如游戏手柄），需要调高带宽。

### Q34: W25Q128 在 OTA 中的角色是什么？为什么不用直接写入 Flash？

W25Q128 作为固件**暂存区**（staging area），下载的固件先存到这里，校验通过后再搬运到 STM32 片内 Flash。原因是：
1. **ESP8266 下载速度不确定：** WiFi 可能断断续续，如果边下载边写片内 Flash，中途断开会导致 Flash 处于不一致状态。
2. **片内 Flash 擦除粒度大：** 最小扇区 16KB，如果下载到一半失败，擦除和重试代价大。
3. **W25Q128 容量大（16MB）：** 可以存多个固件版本，4KB 扇区擦除快，适合频繁写入。
4. **下载和升级解耦：** 可以先下载完、校验通过，再决定是否升级。

## 八、UI 与应用层

### Q35: UI 页面管理是怎么设计的？

采用页面枚举 + 函数分发的模式。`PageState_t` 枚举定义所有页面，`UI_Manager` 维护当前页面状态。每个页面模块实现两个接口：`Page_*_Draw()`（绘制）和 `Page_*_HandleKey(key)`（按键处理）。`UI_Refresh` 和 `UI_HandleKey` 通过 switch-case 分发到当前页面的对应函数。切换页面时保存 `LastPage` 并调用页面重置函数。

### Q36: 游戏页面的帧率是怎么控制的？

DisplayTask 的基础刷新周期是 50ms（20fps），但不同页面可以有不同的刷新间隔。Game 页面用 `GAME_UPDATE_PERIOD = 20ms`（50fps），通过 `app_config.h` 中的 `DISPLAY_REFRESH_GAME` 配置。游戏逻辑更新和渲染在同一任务中同步执行，避免画面撕裂。

### Q37: 水平仪功能是怎么实现的？

利用 MPU6050 融合后的 Roll/Pitch 角度，在 OLED 上绘制一个气泡位置。Roll 对应水平偏移，Pitch 对应垂直偏移，角度越大偏移越远。当 Roll 和 Pitch 都接近 0° 时，表示物体水平放置。页面以 200Hz 采样（自适应策略自动切换），确保气泡跟随实时。

## 九、调试与工程

### Q38: 项目中遇到过哪些 Bug？怎么解决的？

1. **MPU6050 初始化偶尔失败：** 原因是上电时序问题，MPU6050 需要等待内部上电完成。解决：写入复位命令后延时 100ms，再读取 WHO_AM_I 寄存器验证。
2. **OLED 显示乱码：** 多任务同时操作 OLED 导致 I2C 总线冲突。解决：引入 OLEDMutex，所有 OLED 操作必须持锁。
3. **计步误触发：** 手臂自然摆动也会触发计步。解决：增加最大阈值（8000）过滤异常冲击，增加最小间隔（200ms）过滤高频抖动。
4. **抬腕检测不灵敏：** 角度阈值设置太严格，正常看表姿势不在范围内。解决：通过实测调整 Roll ±45°、Pitch -45°~+10° 的窗口。

### Q39: 如果让你优化这个项目，你会怎么做？

1. **性能：** MPU6050 数据读取改为 burst read + DMA，减少 CPU 等待。
2. **功耗：** 使能 Tickless Idle，实现 STM32 Stop 模式 + RTC 闹钟唤醒，深度睡眠功耗可降至 ~20μA。
3. **通信：** ESP8266 AT 指令驱动改为中断驱动 + 环形缓冲区，当前轮询方式浪费 CPU。
4. **OTA：** 增加断点续传和固件加密签名，防止降级和中间人攻击。
5. **算法：** 计步算法增加频率分析（FFT 或自相关），提高复杂运动场景下的准确率。

### Q40: 项目中最大的技术挑战是什么？你是怎么解决的？

OTA 方案的设计。因为涉及 Bootloader、App、外部 Flash、EEPROM 四个组件的协调，任何一个环节出错都可能导致设备变砖。通过以下方式保证可靠性：
- A/B 分区保证升级失败不影响当前系统
- 双重 CRC 校验（下载后 + 搬运后）保证数据完整性
- BootFlags 的 magic 字段防止读取到随机数据
- 栈指针校验防止跳转到损坏的 App

## 十、扩展问题

### Q41: 如果要加 BLE 功能，架构上需要怎么改？

1. **硬件：** 增加 BLE 模块（如 nRF52832），通过 UART 或 SPI 与 STM32 通信。
2. **任务层：** 增加一个 BLE 任务（优先级与 KeyProcessTask 相当），处理 BLE 事件。
3. **通信：** 设计 GATT Profile，定义手表数据（步数、心率、通知等）的特征值。
4. **OTA：** BLE 也可以作为 OTA 的传输通道，但带宽比 WiFi 低（BLE 5.0 约 2Mbps），适合小固件。
5. **功耗：** BLE 比 WiFi 省电得多，如果只做 BLE 不用 WiFi，可以去掉 ESP8266，整体功耗大幅降低。

### Q42: 如果要加心率检测，你会怎么做？

1. **传感器：** 增加 PPG 光电传感器（如 MAX30102），I2C 接口。
2. **采集：** 在 MPU6050Task 中同步读取 PPG 数据，或新建独立任务。
3. **算法：** 对 PPG 信号做带通滤波（0.5-5Hz，对应 30-300BPM），然后峰值检测计算心率。可以结合加速度数据做运动伪影消除。
4. **显示：** 在 UI 中增加心率页面，实时显示 BPM 和波形。

### Q43: FreeRTOS 的任务调度具体是怎么工作的？

FreeRTOS 使用抢占式调度 + 同优先级时间片轮转：
- **抢占式：** 高优先级任务就绪时立即抢占低优先级任务。比如 KeyProcessTask(3) 就绪时会抢占 DisplayTask(2)。
- **时间片轮转：** 同优先级任务轮流执行，每个任务运行一个 tick（1ms）。配置了 `configUSE_TIME_SLICING = 1`。
- **就绪列表：** 用 `configUSE_PORT_OPTIMISED_TASK_SELECTION = 1`，ARM 的 CLZ 指令可以在 O(1) 时间找到最高优先级的就绪任务。
- **阻塞/挂起：** 任务调用 `vTaskDelay`、`xQueueReceive` 等会进入阻塞态，让出 CPU。如果没有就绪任务，调度器运行 Idle Task（执行 WFI）。

### Q44: 中断优先级和 FreeRTOS 优先级有什么关系？

两者是完全不同的概念：
- **FreeRTOS 任务优先级：** 软件层面，数字越大优先级越高，范围 0-7。
- **NVIC 中断优先级：** 硬件层面，数字越小优先级越高，STM32F407 有 4 位（16 级）。
- **关系：** `configLIBRARY_MAX_SYSCALL_INTERRUPT_PRIORITY = 5`，意味着优先级 0-4 的中断**不能调用 FreeRTOS API**（如 `xQueueSendFromISR`），优先级 5-15 的中断可以。这是为了保护 FreeRTOS 内核的关键段不被更高优先级的中断打断。

### Q45: 如果系统运行一段时间后 FreeRTOS 堆耗尽了怎么办？

1. **预防：** 配置了 `configUSE_MALLOC_FAILED_HOOK = 1`，分配失败时会进入 `vApplicationMallocFailedHook` 打印错误。
2. **监控：** `HEAP_WARNING_THRESHOLD = 1024`，运行时检查剩余堆空间，低于阈值报警。
3. **排查：** 用 `uxTaskGetStackHighWaterMark` 检查各任务栈使用率，确认是栈溢出还是堆碎片化。
4. **优化：** 关键任务改用静态分配（`xTaskCreateStatic`），减少动态分配。如果 heap_4 碎片严重，考虑用 heap_5 支持非连续内存区域。

## 十一、开放性问题

### Q46: 你从这个项目中学到了什么？

1. **系统思维：** 嵌入式开发不只是写驱动，要考虑功耗、可靠性、可维护性等系统级问题。OTA 方案让我理解了"不会变砖"的重要性。
2. **RTOS 设计：** 任务划分、优先级设计、资源共享不是拍脑袋决定的，要根据实时性要求和数据流方向来规划。
3. **算法调参：** 卡尔曼滤波和计步算法的参数都需要实际测试来调整，理论模型和实际场景有差距。
4. **调试能力：** 嵌入式调试不像 Web 开发有浏览器 DevTools，需要善用串口打印、逻辑分析仪、断点调试等手段。

### Q47: 如果重新做这个项目，你会有什么不同的做法？

1. 从一开始就设计好 OTA 分区布局，而不是后来才加。
2. 用硬件 I2C 替代软件 I2C，减少 CPU 占用。
3. 用中断驱动 + DMA 方式读取 MPU6050，而不是轮询。
4. 使能 Tickless Idle，做更深度的低功耗优化。
5. 加 BLE 代替 WiFi，更适合穿戴场景。
6. 写单元测试，特别是计步算法和卡尔曼滤波器，方便验证参数调整效果。
